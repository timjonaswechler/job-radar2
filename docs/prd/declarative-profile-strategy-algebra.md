# Declarative Profile Strategy Algebra and Source Specialization

## Problem Statement

Job Radar must integrate recruiting systems and career-site families whose detection signals, public endpoints, data formats, pagination models, and posting pages differ substantially. Greenhouse, Workday, and SAP SuccessFactors already demonstrate that this variance cannot be handled reliably by treating URL structure as the primary integration contract or by adding provider-specific backend branches.

For users, adding a Source should require as little integration knowledge as possible. A user should be able to submit an entry point, receive an evidence-backed Source Proposal with an automatically completed Source Config and selected Access Path, and validate the concrete Source with a Source Live Check.

For profile authors, reusable behavior should be assembled from a common declarative vocabulary rather than implemented as ATS-specific Rust. A Source must also be able to specialize any execution-relevant part of its selected Source Profile directly in the same nested document shape. This is necessary because two Sources belonging to the same recruiting-system family can expose different feeds, selectors, bounds, or extraction behavior.

The current Profile DSL already has useful generic capabilities, but Detection, `discovery`, and `detail` now use the final Policy-bearing Strategy Set contract. Schema-v3 Direct Source Specialization now uses typed keyed fragments at the Source root. Discovery also assumes that normalized title and company data are immediately available, although some Sources initially expose only URLs or other inexpensive hints. This can cause guessed URL-derived values to be treated as canonical posting data.

Without a clearer algebra and compilation model, further catalog work risks becoming a collection of increasingly specific URL patterns, selectors, and backend exceptions rather than a reusable integration system.

## Solution

Introduce schema version 3 of the declarative Source Profile and Source model around three consistently named phases:

- `detection` identifies a reusable Source Profile and produces an actionable Source Proposal;
- `discovery` discovers posting references, provider values, or inexpensive search hints;
- `detail` loads fields for one posting occurrence lazily when a caller requires them.

Each phase owns a Strategy Set. A Strategy Set contains a Strategy Policy, bounded Strategies, phase-level acceptance, and a typed phase output contract. Strategies continue to compose reusable acquisition and interpretation primitives such as fetch, pagination, parse, select, predicates, captures, extraction, transforms, acceptance, and diagnostics. The Profile Compiler verifies that every primitive has the required phase context, that all network and browser behavior is bounded, and that accepted outputs satisfy the phase contract.

Schema v3 Parse admits exactly `json`, `xml`, and `html`. The Profile Compiler turns each authored variant into an immutable typed Parse plan whose expected input is either complete strictly decoded HTTP text or complete Browser-rendered text. These input kinds remain distinct and Browser output is never fabricated as an HTTP response. The three canonical implementations produce one shared Parsed Document family. Malformed input produces one bounded typed failure and no partial Parsed Document; acquisition, decode, budget, or Cancellation terminals cannot invoke Parse because they produce no complete typed input. `text` is not an authored, Serde, compiled, or executable Parse variant. Select and selected-item semantics remain a separate primitive responsibility.

### Canonical Template grammar

Template is parity-owned compiler infrastructure, not an independently registered or dispatched Primitive. One context-neutral compiler turns authored text into immutable literal/reference segments; runtime rendering receives only that compiled plan and an explicit typed value view. Compiled plans, Diagnostics, provenance, and fingerprints never retain resolved values or complete runtime contexts.

References use `{{namespace:key}}`. Each authored placement supplies the exact namespaces and keys available there; Detection additionally admits the bare `{{inputUrl}}` key where applicable. Dot aliases and transform pipes are invalid. Whitespace immediately inside delimiters is ignored, while namespace and key tokens cannot contain whitespace or additional separators. Unknown or unavailable namespaces and keys, empty or nested references, and unmatched opening or closing delimiters are compiler errors before external work starts.

Delimiter escaping is deterministic: `{{{{` emits literal `{{` and `}}}}` emits literal `}}`. Single `{` or `}` characters are ordinary literals. Template transforms remain explicit `transforms[]` behavior rather than pipe syntax.

### Canonical authored HTTP Fetch

HTTP Fetch admits only `GET` and `POST`; an omitted method compiles to concrete `GET`. `GET` has no body, while `POST` may use one `json`, `text`, or `form` body. Every JSON string leaf, text body, form value, URL, and header value is compiled through the canonical Template grammar; non-string JSON values remain literals. HTTP `timeoutMs` is required and must be within `1..=60000` in both direct Serde and schema validation.

Authored headers use exactly the lowercase public names `accept`, `accept-language`, `content-type`, `referer`, `user-agent`, and `x-requested-with`, stored in deterministic order. Credential/cookie/secret-like headers or body keys and invalid rendered header values reject before I/O. Pagination supplies only a typed overlay: target-URL overrides must be absolute HTTP(S) URLs without userinfo, query values are URL-encoded, and JSON-body values may replace existing top-level keys only on a POST JSON body. An explicit HTTP Direct Source fragment uses the same 60,000 ms schema ceiling; a mode-omitted partial timeout remains mode-ambiguous until Effective Source Profile compilation, where the canonical HTTP ceiling is enforced.

One compiled HTTP Fetch owner renders the fully typed phase-neutral H01 request and invokes H01 with the phase-owned allowance/Cancellation context. H01 preserves the numeric response status; the HTTP Fetch boundary projects non-2xx as a distinct safe status failure and exposes no body to Parse. It does not own transport, response buffering, strict decoding, Browser Fetch, phase acceptance, or Detection activation. Profile-DSL Retry remains unsupported and is absent from authored, compiled, runtime, report, and fingerprint surfaces.

Sources remain separate concrete documents. When a Source selects a Profile Access Path, it may place partial Source Profile fragments directly on its root using the same nested field names as the Source Profile. These fragments use dedicated typed Rust documents and a matching JSON Schema `$def`; only execution-relevant fields are representable. Profile identity, `detection`, Search Request fields, persistence fields, and Source Config Schema `title` annotations are impossible to author through this model. `title` is reusable Source Profile authoring metadata that may be inherited into the Effective Source Profile but is not directly Source-specializable. No `overrides` or `overlay` wrapper and no operation/path/value patch language is exposed to authors. The compiler structurally merges the referenced base profile with the direct Source fragments, validates the resulting Effective Source Profile, validates Source Config against its effective schema, resolves the selected Access Path, and compiles one immutable Execution Plan.

The direct Source specialization supports both partial changes to existing keyed Access Paths and Strategies and complete additions of new Access Paths or Strategies. Profile Detection, profile identity, Search Request criteria, and persistence policy do not become source-specializable profile fragments.

The runtime executes only compiled plans and does not know whether behavior originated in a Built-in Source Profile, Custom Source Profile, Source specialization, or Source-owned Access Path. Detail receives one posting occurrence plus an explicit typed set of required canonical fields; one bounded Strategy response may satisfy several requested fields and returns only requested available values.

Search Runs resolve candidates through a data-driven, bounded state machine rather than one fixed Discovery-to-Detail sequence. They evaluate available provider values and explicitly authorized `search_prefilter` hints, reject or finalize when possible, request only missing required fields, and re-evaluate after enrichment. Discovery is consumed in deterministic bounded batches. Search Runs remain responsible for normalization, final Match Rules and Exclusion Rules, cross-Source deduplication, budget-/ceiling-based partial completion, and persistence of finalized Matches only. Cancellation retains the existing Search Run cancellation behavior and is not represented as persistable Resolution Partial Completion.

## Implementation Decisions

1. Schema version 3 uses the phase names `detection`, `discovery`, and `detail` consistently across authored documents, compiled plans, diagnostics, and domain-facing interfaces.
2. A phase exposes a Strategy Set consisting of a Strategy Policy, bounded Strategies, phase-level acceptance, and a typed output contract.
3. Initial Strategy Policies are `first_accepted`, `all_required`, `collect_all`, and `at_least`. Additional policies require evidence and precise cardinality semantics.
4. Strategy Policies operate on accepted Strategy attempts. Transport success alone does not mean Strategy acceptance.
5. Predicate composition such as all, any, none, negation, equality, containment, regex, and count requirements is distinct from Strategy Policy composition.
6. Strategies retain a constrained declarative data-acquisition shape rather than becoming arbitrary chains or scripts. Their available operations include acquisition, pagination, parsing, selection, predicates, captures, extraction, transforms, acceptance, and diagnostics.
7. Every phase uses the shared primitive catalog only when the required typed context is available and the output contributes to that phase contract.
8. Detection receives an entry point and bounded probe responses and produces evidence, captures, Source Config contributions, and an Access Path recommendation for one Source Proposal.
9. Discovery receives a concrete Source and compiled Access Path and produces posting occurrences containing stable references, explicit provider values, or clearly marked hints.
10. Detail receives one posting occurrence and an explicit typed `requiredFields` set. It returns a non-conflicting partial field patch containing only requested available values. A single bounded Strategy response may satisfy several requested fields. It remains lazy and is invoked only when a caller needs fields that are not already available.
11. Search Request criteria, final matching, exclusion semantics, location filtering, cross-Source deduplication, persistence, and Source lifecycle transitions remain backend-owned product behavior rather than Profile DSL primitives.
12. A Source that selects a Profile Access Path may place partial execution-relevant Source Profile fragments directly on its root. No `overrides` or `overlay` wrapper is introduced.
13. Direct Source specialization uses a dedicated typed partial-fragment document model plus a matching JSON Schema `$def`. It may cover the executable/validation members of Source Config Schema, Access Paths, Policies, Acceptance, existing and new Strategies, and all safe Strategy primitives. Only execution-relevant fields are admitted; `detection`, profile identity, Search Request fields, persistence fields, and Source Config Schema `title` are unrepresentable. A `title` authored on a reusable Source Profile property is inherited unchanged into the Effective Source Profile; a Source fragment and a Source-owned Access Path may neither add nor replace it.
14. Profile Detection is not source-specializable because it runs before a concrete Source and its specialization exist.
15. Profile document identity is not source-specializable. Source identity remains the meaning of the Source document's schema version, key, name, and status.
16. Access Paths and Strategies are merged by stable key. Existing keyed entries are recursively merged; new keyed entries must become complete valid definitions.
17. Existing keyed entries preserve their base-profile order. New keyed entries append by default until explicit placement or reordering semantics are accepted.
18. Arrays without stable keyed identities are replaced as whole values rather than merged by index.
19. `null` does not implicitly delete inherited behavior. Removal or disabling requires a future explicit typed contract.
20. After structural specialization, the complete Effective Source Profile is validated before Source Config validation and Access Path resolution.
21. The compilation sequence is: load base profile, merge direct Source fragments, validate the Effective Source Profile, validate Source Config, resolve the selected Access Path, and compile the Execution Plan.
22. The runtime executes only an immutable typed Execution Plan and does not inspect raw Source Profile or Source JSON.
23. The runtime does not branch on ATS, profile key, host, company, or Source key.
24. Phase reducers are backend-owned and deterministic: Detection merges evidence and equal captures, Discovery unions occurrences by Source-local occurrence identity, and Detail merges non-conflicting field patches. Source-local occurrence identity is concrete Source key plus an explicit stable provider posting ID, or concrete Source key plus normalized absolute posting URL when no provider ID exists. Title, company, and location do not participate in this identity.
25. Conflicting non-empty captures, Source Config contributions, or canonical field values produce Structured Diagnostics. Implicit last-write-wins behavior is prohibited.
26. Recovered failed attempts remain observable but do not make a later accepted `first_accepted` result fail solely because fallback was exercised. Discovery and Detail execute mandatory compiled Policies through one crate-private typed ordered-attempt kernel. The kernel owns attempt identity/history and satisfied, policy-unsatisfied, cancelled, or stopped terminal selection; phase adapters retain typed inputs and outputs, acceptance, failure classification, reducers, validation, and public Diagnostic projection. HTTP, Browser, and pagination cancellation propagates as typed control state and is translated to a Structured Diagnostic only by the matching phase adapter; Diagnostic codes and text never drive runtime control flow.
26a. `{ "type": "all_required" }` executes Strategies sequentially in compiled order. Every completed Strategy supplies its canonical per-Strategy acceptance fact exactly once; the first rejection or ordinary execution failure stops later debit, effects, acceptance, and reduction. Only universal acceptance invokes the phase reducer once over all accepted outputs in Strategy order. Reducer conflicts do not retroactively reject the Policy. Rejection or failure exposes no accepted prefix and appends exactly one safe `strategy_policy_all_required_unsatisfied` runtime Diagnostic at `/discovery/policy` or `/detail/policy`; budget exhaustion and typed Cancellation take precedence and append neither that Diagnostic nor `fallback_exhausted`.
26b. `{ "type": "at_least", "count": N }` requires a positive `N` no greater than the final merged Strategy cardinality. Strategies execute in compiled order until the Nth canonical per-Strategy acceptance or until `accepted + remaining < N`; equality remains reachable. Threshold success reduces exactly the accepted prefix once in Strategy order, while impossibility exposes no prefix and performs no reduction. Impossibility appends exactly one category `runtime`, severity `error` Diagnostic with code `strategy_policy_at_least_unsatisfied`, message `at_least policy was not satisfied`, path `/discovery/policy` or `/detail/policy`, no `strategy_key`, and details exactly `{ "policy": "at_least", "requiredAccepted": N }`. It exposes no accepted, remaining, or attempted counts, outcome reason, provider data, URL, body, config, or secret; budget exhaustion and typed Cancellation take precedence and append neither this Diagnostic nor `fallback_exhausted`.
27. Every public Discovery or Detail invocation owns one private cumulative allowance root shared by all Strategies and nested work. The immutable backend ceilings are 50 Strategy attempts, 1,000 logical requests, 100,000 produced items, 120,000 milliseconds, 1,000 pages, 50 Browser actions, 100,000 fan-out admissions, and 67,108,864 HTTP response bytes. An optional eight-field `limits` object on a Discovery or Detail Strategy Set may tighten all eight values; Direct Source Specialization and optional caller limits may tighten inherited values further, but zero, `null`, omission/removal of inherited fields, unknown fields, and any raise are rejected rather than clamped. Successful compilation stores mandatory resolved limits in the immutable Execution Plan. Profile DSL Retry is not a phase allowance dimension and remains unsupported.
28. One checked atomic debit evaluates affected dimensions in attempts, requests, produced items, response bytes, duration, pages, Browser actions, and fan-out order before the effect. Equality followed by EOF succeeds; denied work is neither charged nor started; accepted, rejected, failed, or cancelled work already performed is never refunded. Produced-item, fan-out, and admitted HTTP-byte prefixes are charged deterministically. HTTP response-byte capacity is cumulative across fallback, pagination, and sitemap requests and never resets per Strategy or request. A stream stops after proving one excess byte, retains and commits at most the remaining allowance, and exposes no response, decoded text, or phase payload on exhaustion. Browser-rendered strings are separate and never charged as HTTP response bytes. Cancellation is checked before debit/effect and is observed by the acquisition client so an admitted prefix is committed exactly once. Source Live Check tightens the cumulative Discovery request allowance to one request for its whole invocation. Global security and runtime ceilings cannot be raised or disabled by a Source specialization.
28a. Every Discovery or Detail terminal after work starts carries exactly one `PhaseExecutionReport` containing the eight exact usage counters and one typed completion: accepted, policy-unsatisfied, budget-exhausted, execution-failed, or cancelled. Budget exhaustion records the denied dimension, requested and remaining amounts, and tied backend/compiled/caller limit sources in that order. A pre-start Source/plan mismatch has no report. A caller-limit mismatch returns an `execution-failed` report with all eight usage counters at zero. Budget, internal failure, and Cancellation expose no phase payload, and accounting is never reconstructed from Diagnostics. One validated Detail patch counts as one produced item regardless of field count.
28b. Discovery and Detail return one shared closed `PhaseOutcome`: `Completed` carries a `PolicyOutcome`, the exact complete R02 report, and ordered safe Diagnostics; `BudgetExhausted` carries only that report and Diagnostics; and `ExecutionFailed` additionally carries a typed execution failure. `PolicyOutcome` is either `Accepted` with the phase-specific reduced O02 payload or payload-free `PolicyUnsatisfied` with the closed typed cause `RejectedOnly` or `IncludesExecutionFailure`. Cancellation remains typed control flow outside the normal outcome in `PhaseCancelled`, with only the exact report and Diagnostics, while pre-start plan/request mismatch has Diagnostics and no report. Every completed Strategy supplies its P11 acceptance fact before Policy transition. Policy success performs the sole O02 reduction, then one final P11 phase acceptance, then the common Cancellation check immediately before commit. Cancellation at that boundary discards all computed domain data. Diagnostics never select terminal variants or reconstruct report facts.
29. Discovery and Detail HTTP work uses one phase-neutral domain seam with a production reqwest adapter and an ordered deterministic scripted adapter. The shared bounded collector preserves numeric status including non-2xx, redirect-final URL, normalized repeated header names with non-lossy values, normalized media type, and exact delivered bytes. Automatic gzip, Brotli, deflate, and zstd response decompression is disabled, so charged bytes are the body-stream bytes. Text reaches the canonical Parse family only after complete strict decoding: every authored, recognized UTF-8/UTF-16 BOM, and HTTP `Content-Type` charset declaration is validated and canonicalized; aliases of one WHATWG identity agree, conflicting or unsupported declarations fail; selection precedence is authored, BOM, HTTP, then UTF-8. Decoding is incremental in bounded chunks without replacement and consumes a compatible BOM from text while retaining it in raw bytes. Diagnostics expose only typed transport kind, method, sanitized origin/status/media and charset facts, and byte counts; they never retain raw URL paths or queries, credentials, headers, cookies, bodies, text, secrets, or raw transport errors. Authored HTTP Fetch configuration and no-Retry semantics remain P09-owned, parse consolidation remains P02-owned, Browser acquisition remains separate, and D02 consumes this transport/decoder seam through P09 while A02 owns productive Detection activation and deletion of the temporary old Detection client. A Detection operation uses its own cumulative immutable 67,108,864-byte backend scope without an authored Detection limit or Discovery/Detail outcome semantics.
30. Parse has exactly three schema-v3 registrations and canonical execution owners: JSON, XML, and HTML. They consume distinct complete decoded-HTTP or Browser-rendered text inputs and produce one shared Parsed Document family. Plain text, URL, JSON-LD, microdata, sitemap, and feed may reuse future parse or selection behavior only after separately evidenced executable contracts; they are not additional Parse registrations in schema v3.
30a. Select has exactly six schema-v3 registrations and canonical compiled owners: `document`, constrained `json_path`, literal `xml_element`, literal-path `xml_text`, parsed `css`, and `sitemap_urls`. They consume the shared Parsed Document without reparsing and produce one deterministic ordered `SelectedSequence<SelectedItem>`; phase-output interpretation remains adapter-owned. JSONPath supports only `$` and simple dot-separated ASCII alphanumeric/underscore members, while CSS syntax is validated by the canonical selector parser before external work. `xml_element` compares the entire authored value as a case-sensitive local name over the current node and descendants in document order. `xml_text` trims its path; `.` or an empty/slash-only path selects the current node; otherwise empty slash segments are discarded, the first segment may identify the current node or a direct child, later segments require direct children, and descendant text nodes are trimmed and joined with one space. `sitemap_urls` is admitted only for XML Discovery sitemap child/posting selector placements, filters normalized non-empty `<loc>` values in document order with an optional compile-time Rust regex, and owns no I/O, queue, allowance, or request accounting. An omitted posting selector selects all normalized non-empty `<loc>` values; an omitted child selector performs no child traversal.
30b. Cardinality has exactly four schema-v3 registrations and canonical compiled owners: `one`, `first`, `optional`, and `all`. A `CompiledCardinality` consumes an ordered `CardinalitySequence<T>` (including P03's `SelectedSequence<SelectedItem>` without conversion) and returns a closed `CardinalityOutcome<T>` whose shape is either `Scalar(Option<T>)` or `Sequence(Vec<T>)`. The readiness-frozen landed matrix is: on an empty sequence, `one`, `first`, and `optional` return `Scalar(None)`, while `all` returns an empty `Sequence`; on one value, the first three return `Scalar(Some(value))`, while `all` returns a one-value `Sequence`; on many values, `first` returns the first scalar and `all` returns every value in source order, while `one` and `optional` fail with `field_cardinality_mismatch` carrying the expected cardinality and actual count. Cardinality performs no selection, conversion, transform, Value lookup/composition, output mapping, or phase-result construction, and the four variants have no phase-specific placement restriction within admitted Field Expressions. Omitted authored field cardinality compiles to `one`; runtime plans always carry the immutable compiled variant and never reinterpret the authored optional value.
30c. Transform has exactly ten canonical snake_case schema-v3 registrations and compiled owners: `trim`, `normalize_whitespace`, `html_to_text`, `url_decode`, `slug_to_title`, `dedupe`, `to_string`, `split`, `join`, and `regex_replace`. The camelCase transform aliases are invalid. `split` authors `trimParts` and `dropEmpty`, both defaulting to false, and rejects an empty separator during compilation. Rust regex syntax is compiled into the immutable `regex_replace` plan before runtime; execution uses replace-all semantics. A pipeline preserves authored order and explicit scalar/sequence shape: element transforms preserve shape, `split` produces an ordered sequence, `dedupe` retains first occurrence, and `join` consumes the complete sequence into one scalar. The canonical pipeline performs no implicit trim, whitespace normalization, or empty-value removal; phase output adapters may still apply their separately owned canonical field normalization after the pipeline.
30d. `to_string` is an observable typed conversion: text remains unchanged; JSON strings are unquoted; JSON numbers and booleans use their canonical JSON scalar text; XML and HTML nodes produce visible text; and JSON null, arrays, and objects fail the complete pipeline without partial output. `url_decode` performs one strict URI percent-decoding pass, preserves literal `+`, decodes valid case-insensitive `%HH`, and fails the complete pipeline on malformed/incomplete escapes or invalid UTF-8. It never emits replacement text. Runtime transform failures contain only bounded transform/value indexes and phase context, not resolved input values.
30e. Value compilation has exactly four typed placements: Discovery capture source, Discovery filter/output, Detail capture source, and Detail match/filter/output. Discovery capture source admits Source, declared Source Config, and the selected item/document but no posting or capture map; Discovery filter/output additionally admits the complete immutable Strategy capture map. Detail capture source admits Source, declared Source Config, the concrete Posting Occurrence, and the Discovery-declared postingMeta key union, but no selected item/document or capture map; Detail match/filter/output additionally admits the selected item/document and complete immutable capture map. `const` is universal. Unknown Source Config, postingMeta, or capture keys and recursively unavailable direct or Template references reject compilation before I/O; an admitted postingMeta key remains optional on a concrete occurrence. Selector-backed Values delegate document compatibility to P03's selected item/document model. Scalar/sequence shape is explicit and P03's `SelectedSequence<SelectedItem>` is retained rather than copied into a second document model.
30f. `combine` and `first_non_empty` candidates recursively use their enclosing Value placement. Value expression depth is limited to 16 with root depth 1, complete Effective Source behavior to 1,024 expression nodes, and each `first_non_empty` to 16 direct candidates. The first-candidate limit is also enforced by authored JSON Schema; all three limits are independently rechecked after Direct Source Specialization merge and for complete Source-owned Access Paths. Exact boundaries pass and one-over rejects with bounded compiler Diagnostics containing actual/maximum and Strategy context without resolved values.
30g. Value has exactly 13 schema-v3 registrations and compiled execution owners: `const`, `template`, `source_config`, `posting_meta`, `capture`, `item_field`, `json_path`, `xml_text`, `xml_element`, `css_text`, `css_attribute`, `combine`, and `first_non_empty`. `const` admits only authored strings, JSON numbers, and booleans. `ListFieldExpression::Single` retains the child shape; `Multiple` evaluates children in authored order and an authored empty `Multiple([])` succeeds as an empty sequence. `combine` requires at least one ordered part, evaluates each child through its complete Cardinality/Transform/normalization pipeline, fails on an empty required part, omits an empty optional part, joins the remaining scalar pieces, and then applies its own Cardinality and transforms. `first_non_empty` has 1–16 scalar candidates and no wrapper Cardinality; candidate `all`, nesting another fallback anywhere in a candidate subtree, or a candidate/wrapper transform pipeline with sequence output is rejected before execution. Candidates execute sequentially in authored order through lookup, conversion, Cardinality, transforms, trimming, and normalization. Missing, empty, or whitespace-only successes continue; numeric `0` and boolean `false` win; hard child failure aborts without executing later candidates. The first non-empty candidate wins, wrapper transforms execute exactly once with no backtracking, and all-empty succeeds with an empty scalar. Fallback performs no I/O, parallelism, Strategy fallback, retry, refetch, or hidden recovery.
30h. Discovery and Detail execute only the canonical compiled Value plans through their placement-specific typed contexts. Source Config, postingMeta, captures, selected values, and provider payloads are supplied ephemerally at evaluation; resolved values are never retained in an Execution Plan, Value descriptor, Diagnostic, report, fingerprint, log, or persistence artifact. Execution Plans retain only authored scalars, declared identifiers/keys, and compiled Template, selector, Cardinality, and Transform plans.
30i. Predicate has exactly three schema-v3 registrations and compiled execution owners: `non_empty`, `regex`, and `equal`. `non_empty` evaluates its typed Value once and is true exactly when the landed Value result contains a non-empty scalar or sequence entry. `regex` accepts a scalar-shaped Value, compiles Rust regex syntax before external work, evaluates the Value once, and treats missing or empty results as false. `equal` evaluates each typed Value operand once and compares the complete landed Value carriers, including their scalar/sequence shape and missing semantics; Detail retains collection zero/one/multiple selection and phase Diagnostics. Runtime plans retain only compiled Values and authored/compiled regex syntax, never evaluated values. Detection `expectStatus`, `contains`, and `htmlContains` remain D02/D03 Strategy options rather than Predicate registrations: status equality delegates to `equal`, while literal response-body/rendered-HTML containment delegates to one non-registered lower-level Predicate-family behavior. Productive legacy Detection activation/deletion remains A02-owned. Predicate behavior is distinct from Strategy Policy and backend-owned Search Request Match and Exclusion Rules; `all`, `any`, `none`, negation, count, and generic containment are not admitted Predicate identities.
30j. Capture has one canonical compiled named-only owner shared by Discovery and Detail. In each authored `captures` object, the outer Capture key is both the output key and the exact Rust-regex named group selected by that rule; another named group, a group named `value`, and unnamed group 0/1 never act as fallback. Invalid regex syntax or absence of the same-named group is a compiler error before I/O. Every rule evaluates its scalar `from` Value against the same immutable complete phase capture-source context, which contains no capture map, and results retain authored order. A regex miss, an unmatched selected optional group, a missing source, or a value empty after boundary `trim` is a typed atomic Capture failure: all independent rules are evaluated deterministically, but no partial outputs are published. Capture preserves internal whitespace and performs no other implicit normalization; authors use explicit Value transforms for additional normalization. Compiled plans retain only ordered keys, compiled Values, and regex structure. Source text and captured values remain ephemeral and never enter Diagnostics, reports, fingerprints, logs, or persistence. Phase adapters may project a successful complete ordered output into their immutable capture view, but Capture itself never mutates or overwrites a destination map. Productive Detection maps and activation remain D01–D03/A02-owned.
31. Provider location strings are preserved losslessly as inputs to central normalization. Profiles do not implement ATS-specific geolocation semantics.
32. Discovery extraction uses four disjoint sections: required `reference { url, providerPostingId? }`, optional `providerValues { title?, company?, locations?, descriptionText? }`, keyed `hints`, and keyed `postingMeta`. Each hint contains `value` and optional `hintUse`; canonical-looking hint keys remain noncanonical. Only `hintUse: search_prefilter` may later reject a candidate early, and no hint or postingMeta value may populate a Provider Value, identity, final matching value, or persistence field.
32a. Posting Occurrence identity is concrete Source key plus the opaque case-sensitive provider posting ID when present, otherwise Source key plus a conservatively normalized absolute HTTP(S) provider URL. Provider URLs trim surrounding ASCII whitespace, require a host, reject userinfo, and allow fragments only with provider-ID identity. URL fallback uses parser serialization for scheme, host/IDNA, default ports, and dot segments while preserving path/trailing slash, percent encoding, and query order/multiplicity. It performs no redirects, query sorting/stripping, trailing-slash rewriting, provider heuristics, or I/O. ID and URL-fallback variants never correlate merely by URL.
32b. Before Candidate Resolution lands, Search Runs admit only occurrences containing provider title and company to the existing `SourceCandidate` path. Incomplete occurrences remain silent, uncounted, unnormalized, unmerged, unpersisted, and do not invoke Detail; later complete occurrences continue normally.
32c. Detail requests are typed, non-empty sets over exactly `title`, `company`, raw provider `locations`, and `descriptionText`; duplicates collapse deterministically. Accepted patches expose only requested, non-empty available fields and never replace the required posting URL. Discovery groups exact Source-local occurrence identities in first-seen order, while Discovery responsibilities and Detail fields merge equal values with complete attempt/item-coordinate provenance and permanently quarantine conflicts. Provider-ID URL conflicts reject the complete group; normalized-URL identities retain only their canonical URL. Location vectors compare atomically without normalization, sorting, union, or deduplication. Public provenance, conflicts, rejections, and reducer Diagnostics expose only logical responsibilities and validated strategy/attempt/item coordinates, never values, URLs, bodies, foreign identities, or Attempt History.
32d. Acceptance has exactly three authored camelCase keys and one canonical compiled owner: `requiredFields`, `minDescriptionLength`, and `minResults`; `maxErrorRatio` is not authored, compiled, or evaluated. Discovery `requiredFields` admits `url`, provider `title`, `company`, `locations`, `descriptionText`, and declared `postingMeta.<key>` values. Detail admits only requested `title`, `company`, `locations`, and `descriptionText`; a rule referencing a field absent from the typed invocation is rejected before I/O. `minDescriptionLength` counts Unicode scalar values in the final normalized `descriptionText`; `minResults` counts accepted reduced Posting Occurrences and is Discovery-only, with every Detail placement rejected during compilation. Phase-level and Strategy-level rules compose deterministically, with duplicate required fields evaluated once and the stricter numeric minimum taking precedence. Each completed Strategy is evaluated over its applicable reduced O01/O02 output before its Policy transition, and Policy success reduces the selected accepted output or ordered accepted outputs once before one distinct final phase evaluation and terminal projection. Unsatisfied Acceptance exposes no phase payload. Acceptance emits only stable phase/Strategy paths, field identities, item indexes, and bounded actual/configured counts; provider values and raw contributions are never retained. Acceptance owns facts only and does not define Strategy Policy, O01 identity/provider semantics, O02 conflict reduction, or O03 terminal envelopes and Cancellation.
33. Source Live Check fingerprints are defined only over the canonical schema-v3 representation. They include the base profile, direct Source specialization, Source Config, selected Access Path, compiler/runtime logic, and relevant global behavior versions. No pre-v3 fingerprint format or subsequent fingerprint migration is introduced.
34. Repeated equivalent Source specializations are permitted. The product may emit a non-blocking promotion suggestion but does not block or automatically rewrite user documents.
35. A Source-owned Access Path continues to use the same phase, Strategy Policy, primitive, compiler, and runtime contracts when no reusable base profile is selected.
36. Delivery is compiler-first: compile the Effective Source Profile while initially producing today's Execution Plan shape, prove runtime parity, make `first_accepted` explicit, move internal phase modules directly to their final names, and then perform one schema-v3 authored hard cut. Shared Primitives, additional policies, Detection convergence, and Search Run finalization follow in vertical slices.
37. Schema version 3 is a pre-production hard cut. No permanent or between-ticket compatibility runtime interprets both old and new authored models. A moved slice updates all callers and deletes replaced files, aliases, forwarding functions, superseded implementation-detail tests, and migration-history names in the same ticket.
38. Existing Strategies retain base order and complete new Strategies append. Authored placement/reordering is deferred. `null`, deletion, and disabling remain invalid. Automatic promotion of repeated specializations, parallel Strategy execution, and resumable batches are deferred.
39. Every authored DSL Primitive has exactly one canonical Rust implementation file under `profile_dsl/primitives/<family>/<dsl_type>.rs`. The file owns its primitive-specific authored configuration, compilation, context/capability validation, bounds/security checks, execution semantics, and narrow edge-case tests as applicable. Shared infrastructure must not hide primitive-specific dispatch.
40. `primitives/mod.rs` is registry/dispatch only. A registry completeness test compares schema/document variants with compiled registrations and rejects missing or duplicate Primitive implementations.
41. Candidate Resolution is backend-owned and data-driven. Evaluation returns rejected, finalizable, needs-fields, unresolved, failed, or budget-skipped. Detail enrichment and evaluation may repeat only within explicit per-candidate and cumulative bounds.
42. Discovery candidates are processed in deterministic bounded batches until exhaustion, cancellation, or a backend-owned ceiling. Budgets cover at least discovered items, batch size, Detail candidates, requests, bytes, retries, pages, duration, and enrichment rounds. Profiles and Sources may tighten but never raise ceilings.
43. Candidate Resolution completion is typed independently of Source Status:

    ```rust
    enum ResolutionCompletion {
        Complete,
        Partial { reason: PartialReason },
    }
    ```

    `Complete` means the Source's Discovery stream was exhausted; individual `unresolved` or `failed` outcomes may still be non-zero. `PartialReason` identifies budget/ceiling exhaustion or another explicitly bounded non-cancellation Source execution stop. Cancellation is not a `PartialReason`: it follows the existing Search Run cancellation path, does not produce persistable Resolution Partial Completion, and does not automatically release already finalized candidates for downstream persistence. Completion carries exact aggregate counts for `discovered`, `processed`, `finalized`, `rejected`, `unresolved`, `failed`, and `budgetSkipped`, plus optional `remaining` only when the provider makes that value known. The invariants are `processed = finalized + rejected + unresolved + failed` and `discovered = processed + budgetSkipped`. Resolution completion is not a new `SourceStatus`, `SourceRunStatus`, or `SearchRunStatus`.
44. On budget-/ceiling-based Partial Completion, already finalized Matches remain usable; unfinalized candidates are not persisted. Candidate-scoped diagnostics, including all Candidate errors, are sampled in deterministic provider order under an immutable backend-owned per-Source-resolution limit. A diagnostic summary retains complete Candidate `countsByCode`, `sampleLimit`, and `candidateDiagnosticsOmitted` values after samples are truncated. Outcome counts and per-code Candidate diagnostic counts are never sampled. Source-level terminal diagnostics needed to explain `PartialReason` are stored separately and are not charged against the Candidate sample limit.
45. `ResolutionCompletion` and `ResolutionCounts` are visible on the corresponding `SourceRunResult` when Candidate Resolution ran. `SearchRunResult` exposes a derived Resolution summary aggregating complete/partial Source Run resolutions and their counts. Cancelled, skipped, or pre-resolution-failed Source Runs may have no Resolution completion. No new `SourceStatus`, `SourceRunStatus`, or `SearchRunStatus` variant is introduced.
46. Cross-Source Job Posting deduplication remains a separate backend rule after provider values are loaded and normalized. Its tolerance is not a Profile Primitive and is not part of `PostingOccurrenceIdentity`.
47. Effective Source Config Schema uses one evidence-backed constrained contract shared by the Profile Compiler and Profile Detection; it is not a general JSON Schema interpreter. The root supports optional scalar `type: "object"`, `properties` as an object, `required` as an array of unique string property names, and boolean `additionalProperties`. After same-location Source specialization and profile/Access-Path composition, every `required` name must identify exactly one declared property in the composed contract; duplicates and undeclared names are invalid schema definitions. Each complete property schema requires one scalar `type` from `string`, `number`, `integer`, `boolean`, `object`, `array`, or `null`. Evidence-backed executable property keywords are Rust-regex `pattern` for strings, a non-empty unique scalar `enum` compatible with the declared type, `format: "uri"` for strings using absolute-URI semantics, and inclusive numeric `minimum` for `number`/`integer`. Reusable Source Profiles may additionally author a non-empty string `title` annotation. `title` is non-executable, is inherited unchanged into the Effective Source Profile, and is forbidden in direct Source fragments and Source-owned Access Paths. Source specialization recursively merges `properties`, requires every new property to become complete, replaces `required` and `enum` as whole non-keyed arrays, and replaces executable scalar keywords; it cannot add or replace `title`. Invalid schema definitions emit `compiler` diagnostics, invalid saved Source Config emits `source_validation` diagnostics, and invalid Detection proposals emit `detection` diagnostics through one shared in-process validator. Unsupported keywords, shapes, keyword/type combinations, duplicate or undeclared `required` names, and unsupported type values are never ignored. Any expansion beyond this subset requires concrete evidence and a separate ticket.
48. Browser Detection uses immutable backend ceilings of one target navigation per browser Strategy, two per profile Strategy Set, and eight per Detection operation; authored action `maxCount` at most five, with at most ten executed actions per profile Strategy Set and 32 per Detection operation; four logical waits per navigation, eight per profile Strategy Set, and 32 per Detection operation, with a 5,000 ms per-wait timeout; browser duration, including teardown, of 20,000 ms per Strategy, 30,000 ms per profile Strategy Set, and 60,000 ms per Detection operation; and rendered UTF-8 HTML of 2 MiB per navigation, 4 MiB per profile Strategy Set, and 16 MiB per Detection operation. Every executed nonzero `waitAfterMs` consumes one logical wait, selector-polling sleeps remain part of their single logical wait, and implicit settling consumes one logical wait and duration. Every reservation/debit is checked and committed atomically against every applicable Strategy, profile Strategy Set, and Detection-operation scope. Authored values above immutable ceilings are compiler errors rather than clamped. Oversized rendered HTML is rejected before exposure or parsing, consumes the remaining applicable byte allowance, and reports only a safe observed size; any unavoidable temporary `page.content()` allocation risk is documented. Duration ceilings include cancellation-aware bounded teardown or forced termination, so cleanup cannot await without bound or permit later work. The final 2,000 ms inside every applicable browser duration ceiling—including the single cumulative deadline of each Browser-backed public Discovery or Detail invocation—is reserved for teardown and partitioned into maximum slices of 500 ms for graceful close, 1,000 ms for forced termination and reap, 250 ms for handler completion or abort, and 250 ms for filesystem finalization; unused time carries forward, page work stops before the reserve, no phase extends the hard deadline, and authored or caller-tightened Browser-capable phase durations below 2,000 ms are rejected. Forced termination synchronously initiates the OS kill before the bounded reap wait; if the OS does not report process reap inside its slice, the phase returns a teardown failure within the hard deadline after aborting and awaiting its handler task, without leaving application tasks or futures running. Redirects are not separate target navigations, absent optional-click targets do not consume actions, and Chrome subresource bytes are not claimed as accounted.
49. T16 is not ready for publication or agent execution until one concrete immutable backend-owned Candidate Diagnostic Sample Limit is proposed and accepted. #57 remains responsible for later structured Location semantics and does not block T16, which uses the current Location model.

The decision-rich prototype model is:

```text
Base Source Profile
  + direct Source profile fragments
  = Effective Source Profile
  → validated Source Config
  → selected Access Path
  → immutable Execution Plan
```

### Module and interface decisions

The Effective Profile Compiler, Strategy Set Runtime, and Candidate Resolution are **Deepening Candidates** until implementation evidence satisfies the deep-module acceptance criteria. Design It Twice compared a minimum interface, maximum justified flexibility, and simplest common-caller interface for each candidate. The selected directions are:

1. **Effective Profile Compiler:** one in-process interface accepts a concrete authoritative Source and immutable `SourceProfileRegistrySnapshot` and returns either one `CompiledSource` or Structured Diagnostics. The compiler may resolve profiles from the snapshot but must never select a second Source from `snapshot.sources`; the directly supplied Source is authoritative. The snapshot is immutable input data, not a port or adapter. For a profile-based Access Path, `CompiledSource` contains the immutable Execution Plan and inspectable `EffectiveSourceProfile` plus later provenance/fingerprint material. The module owns merge and validation order; Source lifecycle admission and persistence remain outside. No registry, merge, or validator adapter is introduced.
2. **Strategy Set Runtime:** public callers use typed phase operations for Detection, Discovery, and Detail. A crate-private typed kernel owns policy execution, attempt history, cumulative budgets, cancellation, provenance, and deterministic stop behavior. Phase adapters own typed inputs, outputs, acceptance, and reducers. HTTP/browser execution has production and deterministic test adapters; pure policy, reducer, parser, selector, and field logic remains in-process.
3. **Candidate Resolution:** Search Run calls one Source-scoped resolution operation, not a public per-candidate loop. The module owns deterministic Discovery batches, candidate states, minimal requested-field Detail rounds, cumulative budgets, cancellation, progress, and visible partial completion. A production Source execution implementation plus deterministic fake establish the execution seam. Cross-Source deduplication and persistence remain downstream; UI-only lazy description loading continues to call candidate-scoped Detail directly.

For each candidate, the implementation ticket must confirm caller knowledge, hidden complexity, error modes, test surface, and the deletion test. If deleting the proposed module would make complexity disappear rather than spread into several callers, it must not be introduced. Pure in-process logic receives no speculative trait or port. Tests move to the selected interface and replace superseded implementation-detail tests.

## Testing Decisions

1. The primary test seam is the public Profile Compiler interface: one authoritative concrete Source plus an immutable registry snapshot containing its base Source Profile produces either one `EffectiveSourceProfile`/Execution Plan or Structured Diagnostics. A conflicting same-key Source inside the snapshot cannot change compilation. The snapshot is input data, not a port.
2. Compiler integration tests verify recursive object merge, scalar replacement, keyed Access Path merge, keyed Strategy merge, new keyed entries, whole-array replacement, and rejection of unsupported deletion semantics.
3. Compiler tests verify that source-specialized Source Config Schema is applied before compilation; `properties` merge recursively; new properties are complete; `required` and `enum` replace as whole arrays; `required` names are unique and resolve to declared composed-contract properties; inherited reusable-profile `title` values survive unchanged while direct Source and Source-owned `title` authoring is rejected; and invalid saved Source Config produces source-validation diagnostics.
4. Compiler tests verify that new source-specific Access Paths can be selected only after the Effective Source Profile has been validated.
5. Compiler tests verify typed phase context: unavailable references fail compilation through public diagnostics rather than private helper assertions.
6. Compiler tests verify cumulative bounds and immutable global ceilings for every Strategy Policy.
7. Strategy Policy tests use deterministic fake Strategies through public execution interfaces and cover accepted counts, stop behavior, cancellation, recovered attempts, policy failure, and stable diagnostics.
8. Detection tests use the public Profile Detection interface with deterministic HTTP and browser clients and assert complete Source Proposals, captures, provenance, conflicts, and selected Access Paths.
9. Discovery tests compile complete Sources and execute plans with deterministic fetchers for JSON, XML, HTML, sitemap, feed, pagination, fallback, and collection merge behavior.
10. Detail tests compile complete Sources and execute candidate-scoped plans with deterministic fetchers, asserting typed requested field sets, one response satisfying several fields, requested-only partial patches, field conflicts, reuse, fallback behavior, and laziness.
11. Candidate Resolution tests use the Source-scoped interface with deterministic Source execution dependencies. They verify deterministic batches, conservative authorized-hint prefiltering, minimal requested-field Detail, cumulative bounds, normalization, repeated final Match/Exclusion/location evaluation, and propagation of Cancellation into the existing Search Run cancellation path rather than Resolution Partial Completion.
12. Downstream Search Run integration tests use a real temporary database. They verify cross-Source deduplication and persistence after finalization, prove that unfinalized, failed, unresolved, and budget-skipped candidates create neither Job Postings nor Matches, show that finalized Matches remain usable when later candidates exhaust a budget/ceiling, and prove that Cancellation does not automatically persist already finalized candidates.
13. Source Live Check tests verify schema-v3 Effective Source Profile, direct Source specialization, and Source Config fingerprints and the resulting fresh/stale transitions; no pre-v3 fingerprint fixture exists.
14. Registry tests verify that Built-in and Custom Source Profiles remain reusable while direct Source specializations remain scoped to one Source.
15. Regression fixtures cover Greenhouse as a simple public API profile, Workday as a paginated POST/JSON profile, and SuccessFactors as a feed/sitemap/HTML profile.
16. Tests assert behavior and diagnostics through public crate interfaces wherever possible. Private helper tests are reserved for narrow parser or merge edge cases that cannot be observed through the public seam.
17. No network-dependent check is added to default CI. Current public behavior remains documented separately through Source Live Check evidence.
18. Migration tests and repository searches prove that schema-v3 documents contain no active schema-v2 phase names, old operation-list Source Override model, compatibility dispatch, migration aliases, forwarding wrappers, or duplicate phase implementations.
19. Schema/Serde parity fixtures prove that the typed direct Source-fragment documents and their JSON Schema definition admit the same fields and reject forbidden identity, `detection`, Search Request, persistence, Source Config Schema `title`, `null`, deletion, and disabling shapes.
20. Registry completeness tests prove that each authored Primitive has exactly one canonical Rust implementation registration and that `primitives/mod.rs` contains no Primitive behavior.
21. Candidate Resolution tests assert exact Discovery batches, Detail calls, requested field sets, typed complete/partial completion, both count invariants, SourceRunResult visibility, derived Search Run Resolution summary, bounded diagnostic samples with `candidateDiagnosticsOmitted`, Cancellation outside Partial Completion, no-progress termination, and deterministic diagnostic order through the Source-scoped interface.

## Out of Scope

- Implementing every candidate primitive listed in the exploratory design catalog.
- Arbitrary scripts, plugins, shell commands, JavaScript evaluation, or unrestricted browser automation in profile documents.
- CAPTCHA bypass, automated login flows, or inline credential storage.
- Moving Search Request include rules, exclusion rules, locations, radius, or other criteria into Source Config or Source Profiles.
- Profile-authored database writes, transaction behavior, Job Posting lifecycle, or Source Status transitions.
- ATS-specific Rust adapters or branches.
- Automatic promotion of repeated Source specializations into Source Profiles or Access Paths.
- Blocking users from repeating the same Source specialization.
- Treating Profile Support Level as current operational confidence.
- Treating deterministic fixtures as proof that a concrete Source currently works.
- Final structured location semantics and exact cross-Source deduplication tolerances; those remain backend-owned and part of focused normalization/location work.
- Any schema-v2 compatibility runtime or committed migration wrapper between tickets.
- Authored Strategy placement/reordering, deletion, disabling, or `null` semantics.
- Unbounded recursive crawling or unrestricted Strategy nesting.
- Parallel Strategy or candidate execution until deterministic merge, cancellation, rate-limit, and cumulative-budget behavior are accepted.
- Resumable Discovery cursors/checkpoints in the first Candidate Resolution release.
- A general JSON Schema interpreter or Source Config Schema keywords beyond the evidence-backed subset in Implementation Decision 47.
- Rich profile-authoring UI beyond what is required to represent and diagnose the accepted document model.

## Further Notes

This specification was synthesized from the working architecture note `docs/profil source algebra refactor.md`, which remains unchanged as a design reference from this point forward.

Issue #33 remains an evidence and catalog audit. Findings from Greenhouse, Workday, and SuccessFactors are acceptance evidence for this specification, not justification for provider-specific runtime behavior.

Issue #57 remains responsible for structured location semantics. This specification requires only that explicit provider values be preserved and distinguished from hints before central normalization.

The decisions needed for implementation ticket decomposition are settled: direct Source fragments use dedicated typed documents plus a matching schema definition; delivery is compiler-first with one clean schema-v3 hard cut; Source-local posting occurrence identity is distinct from cross-Source Job Posting deduplication; Detail receives typed requested field sets; Candidate Resolution is data-driven and batch-bounded; and every authored Primitive has one canonical Rust implementation file. Strategy placement/reordering, deletion/disabling, automatic specialization promotion, parallel execution, and resumable batches are explicit deferrals rather than blockers.
