# Research: Issue #44 — browser-assisted element picker

## Summary

Orca implements a complete Electron “grab mode” around an already-managed browser view: typed cross-process messages, an injected guest-page interaction script, main-process session ownership, payload construction, screenshot capture, browser-manager integration, and a React hook. Job Radar should reuse that *protocol and lifecycle shape*, not its Electron/WebContents implementation: a Tauri command should create a bounded picker session, `chromiumoxide` should inject a fixed application-owned script and receive a small typed result over CDP, Rust should validate it, and the frontend should receive state/result events.

The picker should initially be an authoring aid that proposes and verifies CSS selectors for a draft Source-owned Access Path or Direct Source Specialization. It must not add arbitrary JavaScript to the Profile DSL, silently activate a Source, persist page content/screenshots by default, or bypass the existing Profile Compiler, Profile Detection, Source Live Check, and Browser Runtime boundaries.

## Findings

1. **Orca is a coordinated feature, not a selector helper.** The shared types define the renderer/main/guest contract; the guest script owns hover highlighting, click selection and cancellation; the session controller owns start/stop and cleanup; payload and screenshot modules normalize the returned artifact; the browser manager connects grab mode to browser lifecycle; and `useGrabMode` presents the asynchronous state machine to React. The useful lesson is to make selection an explicit, cancellable session with one owner and deterministic teardown, rather than exposing a generic “evaluate JavaScript” command. [types](https://github.com/stablyai/orca/blob/39b124c75bc3b128028ca9c862a5f67e67592907/src/shared/browser-grab-types.ts#L87-L128) · [guest script](https://github.com/stablyai/orca/blob/39b124c75bc3b128028ca9c862a5f67e67592907/src/main/browser/grab-guest-script.ts#L668-L839) · [session controller](https://github.com/stablyai/orca/blob/39b124c75bc3b128028ca9c862a5f67e67592907/src/main/browser/browser-grab-session-controller.ts#L82-L222) · [React hook](https://github.com/stablyai/orca/blob/39b124c75bc3b128028ca9c862a5f67e67592907/src/renderer/src/components/browser-pane/useGrabMode.ts#L13-L217)

2. **The reusable parts are conceptual and protocol-level.** Reuse: typed session/result/error states; one active session per page; hover/selected visual states; explicit cancel (including Escape); payload size/shape validation; page/session identity checks; screenshot as an optional derivative; cleanup on navigation, close, cancellation and completion; and a small frontend hook/state machine. Do not transplant Electron-specific IPC, `WebContents`/guest APIs, BrowserView ownership, main/renderer channel wiring, or Electron image primitives. Orca’s separation of payload and screenshot work is especially useful because Job Radar can independently decide whether either artifact crosses or persists beyond the Rust boundary. [payload validation](https://github.com/stablyai/orca/blob/39b124c75bc3b128028ca9c862a5f67e67592907/src/main/browser/browser-grab-payload.ts#L18-L118) · [screenshot isolation and bounds](https://github.com/stablyai/orca/blob/39b124c75bc3b128028ca9c862a5f67e67592907/src/main/browser/browser-grab-screenshot.ts#L26-L105) · [browser-manager boundary](https://github.com/stablyai/orca/blob/39b124c75bc3b128028ca9c862a5f67e67592907/src/main/browser/browser-manager.ts#L1598-L1662)

3. **The Job Radar flow should stay behind its managed-browser abstraction.** Recommended flow: (a) React asks Rust to start a picker for a URL and field purpose; (b) Rust creates and owns a visible, ephemeral managed Browser Runtime session and returns an opaque session/page capability; (c) the browser adapter uses CDP through `chromiumoxide` to install a fixed, versioned picker script and result binding; (d) the script draws an in-page overlay and returns only a bounded candidate payload (selector candidates, tag, safe attributes, short normalized text fingerprint, bounding rectangle, frame information); (e) Rust rejects stale/oversized/malformed results, re-runs selectors in the same document to record uniqueness, and ends the session; (f) Tauri emits typed state/result events; (g) React shows a proposal that the user must accept; and (h) normal schema/compiler/live-check paths validate the resulting authored document. CDP operations are implementation details of the browser adapter, not Profile DSL capabilities. This follows ADR 0003’s decision that the pinned Chrome for Testing installation and `chromiumoxide` remain behind a Job Radar abstraction. [ADR 0003](../adr/0003-managed-browser-runtime.md)

   **Current gap:** Job Radar does not yet own a persistent interactive page. The current browser client launches Chromium, renders one request to HTML, and tears the process/session down. Therefore session/page ownership and headed-browser control are prerequisites, not existing UI plumbing.

4. **The picker belongs at an authoring boundary, not inside runtime execution or Profile Detection.** Profile Detection deterministically produces a Source Proposal from declared checks and may itself use bounded browser probes. The picker instead captures a user’s intent while authoring extraction rules. Its output may populate a draft CSS `select`/`extract` expression in a Source-owned Access Path or Direct Source Specialization, but the Profile Compiler remains authoritative and Source Live Check provides operational evidence. A click must not be treated as detection evidence, a successful live check, or permission to activate a Source. The Profile DSL explicitly prohibits arbitrary JavaScript and requires browser interactions, waits and network work to be bounded. [Profile DSL PRD](../prd/declarative-source-profile-dsl.md)

5. **Selector generation must optimize for reproducibility, not shortest syntax.** Generate several ranked candidates and verify each against the current document. Prefer a unique, syntactically valid ID only when it does not look generated; then stable semantic attributes (`data-*`, `name`, `role` plus label where appropriate); then stable class subsets anchored to a nearby stable container; use structural `:nth-of-type()` only as a last-resort discriminator. Escape identifiers correctly. Store the chosen CSS selector plus non-executable evidence (match count, tag, selected stable attributes, short text fingerprint and optional ancestor fingerprint), not a DOM path alone. Warn when the selector matches zero/multiple nodes, contains generated-looking tokens, depends on text/localization, crosses an iframe, or targets an open shadow root. Closed shadow roots and cross-origin frames should be explicitly unsupported in the MVP. Re-verify on a fresh navigation and through Source Live Check before presenting the authored Source as usable.

6. **Security and privacy require stricter defaults than a general browser inspector.** Inject only bundled, hash/version-identifiable application code; expose no general evaluate/eval API to the renderer or authored JSON; accept results only for the active opaque session and expected page/frame; enforce URL scheme, payload byte, text length, attribute allowlist, timeout and screenshot bounds in Rust; and remove bindings/overlay on every terminal path. Do not collect input values, password fields, cookies, storage, authorization data, full HTML, or network credentials. Do not persist screenshots or selected text by default; if a screenshot is later offered, require explicit user action, crop it, show what will be saved, and provide immediate deletion. Logs and diagnostics should contain selector/session metadata, not page content or query-string secrets. Navigation or origin change must cancel the session.

7. **Orca’s MIT code may inform or be adapted only with attribution discipline.** Concepts such as session ownership and typed state machines are not copyrightable, but copying or closely adapting the guest script, payload logic, or selector algorithm requires preserving the Orca MIT license and copyright notice in the distributed source/license notices. The implementation PR should identify adapted files and the pinned upstream commit. A clean reimplementation still benefits from a design-credit note, but must not falsely imply that Orca code is bundled. Legal/packaging review should confirm where third-party notices appear in the Tauri distribution before copied code lands. [Orca repository at reviewed commit](https://github.com/stablyai/orca/tree/39b124c75bc3b128028ca9c862a5f67e67592907)

8. **Concrete MVP: one visible page, one field, one verified selector.** Scope the first slice to one Rust-owned ephemeral headed-browser page; a single active top-level-document session; hover outline; click/select; Escape/cancel; selector candidate generation; same-document uniqueness verification; typed Rust result; React confirmation UI; and insertion into a *draft* authoring form. Exclude screenshots, raw HTML capture, multi-element/list selection, iframe/shadow traversal, persisted browser profiles/cookies, autonomous inference of the whole profile, and automatic Source activation. Unit-test selector ranking and payload validation with fixtures; integration-test session cancellation/teardown through a deterministic local page; manually smoke-test the packaged managed browser.

9. **Phased plan.** **Phase 0 — decisions/contracts:** settle the destination authoring model, typed protocol, privacy defaults, diagnostics, and attribution; resolve formal blocker #57. **Phase 1 — backend spike:** fixed script injection/result binding behind the browser abstraction, one-session ownership, cancellation/navigation cleanup, strict validation, no frontend persistence. **Phase 2 — MVP UI:** overlay, field-purpose prompt, candidate explanation, explicit accept/retry/cancel, draft form integration. **Phase 3 — verification:** fresh-navigation recheck, Source Live Check integration, saved-HTML regression fixtures, accessibility/keyboard behavior. **Phase 4 — optional expansion:** multi-field guided authoring, controlled iframe/open-shadow support and opt-in clipped screenshots only after privacy review. Each phase should remain generic DSL tooling; no ATS-specific Rust branch.

10. **Blockers and sequencing are material.** GitHub currently records open #57 as #44’s formal blocker. That dependency should be reviewed: #57 concerns location normalization and a guided Source-setup walkthrough, so it need not block a narrowly scoped selector-picker spike unless the picker’s first destination is explicitly that walkthrough. The active browser replacement chain—#264, #265, #266, #268, #271, and the final activation cut #272—should land or freeze a stable interactive-session seam first; otherwise the picker will bind to transient `chromiumoxide` ownership/session/event APIs and create a second lifecycle model. Issue #44 should depend on the post-refactor browser abstraction rather than merge parallel browser-manager logic. ADR 0003 covers installation/runtime ownership but does not yet decide interactive page/session ownership, headed control, CDP binding/event transport, or authoring-session persistence.

## Recommended protocol sketch

```text
PickerRequest {
  session_id, page_id, purpose, timeout_ms,
  allow_screenshot: false
}
PickerState = starting | active | selected | cancelled | failed
PickerResult {
  session_id, page_id, document_url_without_fragment,
  selector_candidates[{ selector, match_count, stability_flags }],
  element{ tag, safe_attributes, text_fingerprint, rect, frame_kind },
  diagnostics[]
}
```

The Rust boundary must impose closed enums, maximum lengths/counts and a protocol version. `page_id` must be an opaque backend-issued capability, not a raw CDP target ID supplied by arbitrary frontend code. The accepted selector is still authored input and must pass schema validation, Profile Compiler validation and a bounded Source Live Check.

## Decisions still required

1. **Authoring destination:** Does MVP edit only a Source-owned Access Path, only Direct Source Specialization, or both? Recommendation: start with the simplest existing draft editor surface; do not invent a third persisted picker document.
2. **Page ownership:** Which post-#264/#266/#271/#272 abstraction owns interactive pages, and can one page be shared safely between preview, picker and live-check tasks?
3. **Protocol transport:** Tauri events versus command completion/streaming, including exactly-once terminal delivery and behavior when the frontend unmounts.
4. **Selector contract:** Which CSS subset is admitted by the Profile Compiler/runtime, and are selector stability flags persisted as authored evidence or shown only during authoring?
5. **Frames and shadow DOM:** Confirm top-level DOM only for MVP and define diagnostics rather than silently producing unusable selectors.
6. **Screenshots/content:** Confirm “no capture or persistence by default”; decide later whether opt-in cropped screenshots provide enough value to justify retention/deletion UX.
7. **Browser state:** Confirm ephemeral context/no login for MVP, consistent with the DSL prohibition on credentials, cookies and login flows.
8. **Licensing:** Decide clean-room reimplementation versus adaptation and the required third-party notice location before copying Orca implementation text.
9. **Activation gate:** Define whether successful fresh-navigation verification is advisory or mandatory before a Source Live Check/activation; recommendation: picker verification is advisory, Source Live Check remains mandatory where activation currently requires it.

## Gaps and residual risks

The research does not resolve #57 or the contracts being changed by #264/#266/#271/#272. Dynamic sites can invalidate even a carefully ranked selector, so fixture tests and same-page uniqueness cannot replace a fresh-navigation check and Source Live Check. Cross-origin iframe and shadow-DOM support may require a richer selector representation than a single CSS string; deliberately excluding them from MVP avoids prematurely changing the DSL. Before implementation, independently verify that the immutable Orca paths above match the sparse checkout layout and record Orca’s exact MIT copyright notice in the implementation review.
