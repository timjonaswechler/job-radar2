# Rust Agent capability gap inventory

## Summary

The current Rust Agent core is a deliberately capability-free, ephemeral conversation module: it already provides a proven provider seam, deterministic provider tests, transactional turn commits, stable conversation identity, model/reasoning selection, streaming validation, rollback for an `Aborted` terminal outcome, and credential containment. It has no caller-controlled cancellation operation and supports none of issue #197’s seven probes end to end. The gaps are related but do not yet imply one accepted roadmap: tool-mediated capabilities need a bounded tool protocol and policy evaluation; faithful persisted continuation needs stable structured state; and context management needs an explicit relationship to that persisted state.

Pi at commit [`dcfe36c79702ec240b146c45f167ab75ecddd205`](https://github.com/earendil-works/pi/commit/dcfe36c79702ec240b146c45f167ab75ecddd205) is a behavioral reference, not a parity target. Its low-level tool loop, hooks, session entries, and compaction algorithms are useful references; its unrestricted coding workspace, shell-oriented defaults, extension system, branching, queues, and full `AgentSession` are optional or unsuitable unless a Job Radar requirement later justifies them.

## Evidence boundary

This inventory compares the seven probes agreed in [issue #197](https://github.com/timjonaswechler/job-radar2/issues/197#issuecomment-4990196739) with the checked-in Rust implementation and its pinned Pi baseline. “Current” means executable code or tests in the repository, not an architectural intention. A **proven seam** is claimed only for `ConversationProvider`: both `OpenAiCodexProvider` and the regularly compiled `ScriptedProvider` satisfy it, and external integration tests cross that same interface. All prospective tool, policy, workspace, context, session, and compaction seams remain hypothetical until their interfaces, variation, and tests exist.

## Reusable current strengths and invariants

1. **Small, provider-neutral conversation core.** `AgentConversation` owns an immutable system prompt, completed transcript, selected Agent Model, effective Reasoning Level, and stable internal conversation ID; the request exposes these through app-owned types. The provider-specific transport is behind `ConversationProvider`. See [`conversation.rs`](../../src-tauri/src/agent/conversation.rs) and the accepted [`ADR 0011`](../adr/0011-minimal-agent-conversation-contract.md).
2. **Transactional turn state.** A turn commits its User/Assistant pair only after a valid terminal completion. Failed, aborted, malformed, or dropped streams commit nothing. The stream’s mutable borrow prevents overlapping turns or mid-turn model mutation. Deterministic coverage exists in [`conversation.rs`](../../src-tauri/tests/agent/conversation.rs).
3. **Proven provider seam and deterministic adapter.** `ScriptedProvider` validates the complete app-owned request, stable conversation ID, ordered scripted turns, and exhaustion without network or secrets; production Codex uses the same provider interface. See [`testing.rs`](../../src-tauri/src/agent/testing.rs) and [`conversation.rs`](../../src-tauri/tests/agent/conversation.rs).
4. **Credential containment and redaction.** Credentials remain behind authentication/provider implementation, are stored outside repositories with private permissions, refreshed under a lock before use, and excluded from public/debug representations. Public failures use fixed app-owned categories. The executable evidence is catalogued in [`agent-credential-containment.md`](../security/agent-credential-containment.md); storage is implemented in [`auth.rs`](../../src-tauri/src/agent/auth.rs).
5. **No ambient Agent capabilities.** `ConversationRequest` contains only prompt, messages, model, reasoning, and conversation identity. There are no tool definitions/calls/results, filesystem handles, shell access, application repositories, network clients, or approval callbacks in the public conversation contract. This absence is a security-positive starting point, not support for the isolation probe.
6. **Debug CLI is not a product shell.** The feature-gated debug binary handles login/logout, model/reasoning selection, prompt streaming, and quit only; release builds are rejected. It exposes no tool, workspace, persistence, or Tauri surface. See [`agent_debug.rs`](../../src-tauri/src/bin/agent_debug.rs) and [`dev-agent-debug.md`](../dev-agent-debug.md).

## Probe-by-probe inventory

### 1. Bounded multi-step tool loop

- **Current Rust support:** The provider event protocol streams only Text and Reasoning blocks and terminates as Completed, Failed, or Aborted. `FinishReason` has only Completed and LengthLimit. Messages have only User/Assistant variants. Therefore the core cannot describe a tool, receive a tool call, execute it, append a structured tool result, or continue the same user turn with another provider request.
- **Missing capability/boundary:** The agreed gap includes registered tools, structured calls/results, a host-controlled continuation loop, and deterministic call/time/cancellation/termination bounds. The exact message/event shapes, result-size policy, handling of malformed calls, and relationship to the current transactional commit rule remain design decisions.
- **Product requirement vs optional Pi behavior:** The product requires bounded multi-step execution and structured results. Pi’s sequential/parallel modes, steering/follow-up queues, dynamic tool introduction, partial updates, termination hints, and custom message extension mechanism are optional. Parallel execution should not be inherited before ordering, approval, and side-effect semantics require it.
- **Pinned Pi reference:** Pi’s [`agent-loop.ts`](https://github.com/earendil-works/pi/blob/dcfe36c79702ec240b146c45f167ab75ecddd205/packages/agent/src/agent-loop.ts) validates arguments, emits tool lifecycle events, turns failures into tool results, appends results to context, and loops until no calls remain. [`types.ts`](https://github.com/earendil-works/pi/blob/dcfe36c79702ec240b146c45f167ab75ecddd205/packages/agent/src/types.ts) defines `AgentTool`, structured results, execution modes, hooks, and events. These files demonstrate mechanics, not Job Radar’s required bounds.
- **Gap relationship:** Policy evaluation, scoped workspace operations, and external-information access all need some bounded invocation protocol. This makes the tool protocol a dependency candidate, while issue #200 remains responsible for accepted sequencing.
- **Verification evidence needed later:** A deterministic provider/tool fixture must demonstrate more than one tool request in one turn, structured result replay, enforced bounds, cancellation, and a final completion without network access. Exact fixture matrices belong to the verification and implementation tickets.

### 2. Capability isolation and approval

- **Current Rust support:** Isolation exists only by omission: the conversation has no capabilities. There is no per-call policy evaluation, allow/deny decision, paused approval state, resumable call, or UI-independent approval event. Authentication interaction is specific to OAuth login and is not evidence of an Agent tool-approval interface.
- **Missing capability/boundary:** The agreed gap is explicit host-owned capabilities, per-call Allow/Deny/RequiresApproval evaluation, and a UI-independent paused state in which approval occurs before execution. Registry shape, approval lifetime, restart behavior, timeout behavior, and the metadata visible to policy remain open. Existing credential-containment and redaction invariants constrain every option.
- **Product requirement vs optional Pi behavior:** No ambient filesystem/network/shell/application access, per-call evaluation, and UI-independent approval are product requirements. Whether policy uses a default-deny rule remains open. Pi’s generic `beforeToolCall` hook can block a call but does not itself model a first-class paused approval lifecycle. Coding-agent permission-gate extensions or unrestricted shell conventions are not product requirements.
- **Pinned Pi reference:** [`types.ts`](https://github.com/earendil-works/pi/blob/dcfe36c79702ec240b146c45f167ab75ecddd205/packages/agent/src/types.ts) defines the `beforeToolCall` pre-execution hook and block result; [`agent-loop.ts`](https://github.com/earendil-works/pi/blob/dcfe36c79702ec240b146c45f167ab75ecddd205/packages/agent/src/agent-loop.ts) applies it after schema validation and before execution. This is useful hook ordering, not an approval-state design.
- **Gap relationship:** Policy can only evaluate a concrete invocation, and scoped filesystem, network, or application-data access must not bypass it. Whether a pending approval survives restart is deliberately unresolved and would connect this gap to session persistence.
- **Verification evidence needed later:** Deterministic evidence must show Allow, Deny, and pause-for-approval paths, no execution before approval, per-call evaluation, cancellation, and redacted outcomes without requiring a UI or network. Exact restart and timeout cases depend on later policy decisions.

### 3. Scoped file workspace

- **Current Rust support:** None. The conversation accepts only prompt text; neither it nor the debug harness accepts a workspace root or file capability. Existing Rust filesystem use is credential storage, not Agent-visible workspace access.
- **Missing capability/boundary:** The agreed gap is an explicit workspace for discovering/reading approved text resources and producing controlled changes or artifacts, with safe failure for path escape or unapproved access. Root/resource representation, path and symlink rules, read/write operations, output semantics, provenance, and concrete size limits remain open. General shell and unrestricted filesystem access remain excluded.
- **Product requirement vs optional Pi behavior:** The roadmap probe requires approved-resource discovery/read and controlled outputs; the two product areas establish relevance without yet selecting their exact operations. Pi’s cwd-wide absolute-path reads, image support, direct write/edit tools, bash fallback, local coding conventions, and arbitrary custom operations are optional and too broad by default.
- **Pinned Pi reference:** Pi’s [`read.ts`](https://github.com/earendil-works/pi/blob/dcfe36c79702ec240b146c45f167ab75ecddd205/packages/coding-agent/src/core/tools/read.ts) is useful for offset/limit, byte/line truncation, cancellation, and operation injection, but its path contract permits relative or absolute paths from cwd. [`write.ts`](https://github.com/earendil-works/pi/blob/dcfe36c79702ec240b146c45f167ab75ecddd205/packages/coding-agent/src/core/tools/write.ts) and [`edit.ts`](https://github.com/earendil-works/pi/blob/dcfe36c79702ec240b146c45f167ab75ecddd205/packages/coding-agent/src/core/tools/edit.ts) are references only for controlled-operation mechanics, not acceptable scope defaults.
- **Gap relationship:** Agent-initiated workspace operations need bounded invocation and policy evaluation. Approved resources may also become inputs to context composition. The relationship between generated artifacts and conversation-session persistence remains an open boundary.
- **Verification evidence needed later:** Deterministic temporary-workspace evidence must show approved discovery/read, controlled output, safe rejection of unapproved access and path escape, enforced bounds, and cancellation. Whether an allowed write also requires explicit approval depends on the accepted policy rather than being assumed here. Local filesystem semantics need no live network acceptance.

### 4. Bounded external information capability

- **Current Rust support:** The Codex adapter performs provider and OAuth HTTP internally, but callers and the model cannot request arbitrary external information. This transport is not a reusable network capability and must not be exposed as one.
- **Missing capability/boundary:** The agreed gap is provenance-bearing external information under approval, size, and time limits. Query/request shape, allowlisting, redirects, request counts, supported content, and whether the implementation composes search, HTTP, or bounded browser behavior remain open. Credential/header containment and redaction are fixed constraints.
- **Product requirement vs optional Pi behavior:** A bounded way to obtain attributable external information may be required for both scenarios. A general URL fetcher, autonomous browser, shell networking, arbitrary headers/cookies, or Pi extension ecosystem is not required. Product requirements should be stated in terms of evidence and bounds, not a chosen transport.
- **Pinned Pi reference:** The generic [`AgentTool` contract](https://github.com/earendil-works/pi/blob/dcfe36c79702ec240b146c45f167ab75ecddd205/packages/agent/src/types.ts) can carry text/image content and structured details, and the loop propagates cancellation. The pinned baseline does not supply a narrow Job Radar external-information policy; new primary-source research and a security decision are required once a concrete transport is selected.
- **Gap relationship:** Agent-initiated external access needs bounded invocation and policy evaluation. Any later persistence or context management must preserve the provenance required by the accepted result shape. No ordering relative to workspace capability is decided here.
- **Verification evidence needed later:** A mock adapter must prove the accepted limits, cancellation, provenance, and redacted failure behavior. After a concrete transport is selected, a small opt-in non-authenticated live check may add an explicit human provenance/usefulness checkpoint and redacted pass/fail output.

### 5. Host-owned prompt and context composition

- **Current Rust support:** The host supplies one immutable undifferentiated `String` system prompt at conversation construction. Completed transcript and current User Message are ordered deterministically, but instructions, runtime context, approved resources, provenance, and domain prompt are not represented as distinguishable layers and cannot be updated between turns.
- **Missing capability/boundary:** The agreed gap is distinguishable, ordered layers for instructions, runtime context, and approved resources, with replaceable domain prompts rather than domain behavior in the Agent core. Layer representation, precedence, update semantics, provenance, delimiters, budgets, and the distinction between resource access and inclusion remain open design questions.
- **Product requirement vs optional Pi behavior:** Distinguishable ordered host-owned layers and replaceable domain prompts are required by the probe. Pi’s exact XML-like tags, cwd injection, coding-assistant default prompt, auto-discovered `AGENTS.md`, skills, tool snippets, and prompt templates are optional formatting/discovery choices.
- **Pinned Pi reference:** [`system-prompt.ts`](https://github.com/earendil-works/pi/blob/dcfe36c79702ec240b146c45f167ab75ecddd205/packages/coding-agent/src/core/system-prompt.ts) orders custom/default prompt, appended prompt, context files, skills, and cwd. Its explicit composition is useful evidence, while its coding-specific content should not be ported. The low-level [`types.ts`](https://github.com/earendil-works/pi/blob/dcfe36c79702ec240b146c45f167ab75ecddd205/packages/agent/src/types.ts) keeps system prompt separate from messages/tools.
- **Gap relationship:** Composition can exist for host instructions before tools, while resource-bearing layers depend on the accepted workspace/external result shapes. Persisted continuation must later define what composition state is required for faithful resume; this inventory does not choose source layers versus rendered snapshots.
- **Verification evidence needed later:** Deterministic evidence through `ScriptedProvider` must show the accepted layer ordering, distinguishability, replacement behavior, and exact provider request without domain-specific branches. Delimiter, provenance, and budget cases depend on the later composition contract.

### 6. Persisted sessions

- **Current Rust support:** None. `AgentConversation` always creates a fresh UUID and empty in-memory transcript; there is no serialize/load/resume interface. Opaque replay metadata exists internally for provider history but has no versioned durable representation. Credential storage is intentionally separate and is not session persistence.
- **Missing capability/boundary:** The agreed gap is a Job Radar-owned versioned JSON/JSONL format that can store, restart, load, and continue structured conversation state. The durable representation of messages, future tool activity, pending approvals, artifacts/work state, model/reasoning choices, context composition, and opaque replay metadata remains undecided, as do migration, corruption, storage, retention, deletion, and privacy policies.
- **Product requirement vs optional Pi behavior:** Store, restart, load, and continue structured state are required. Pi file compatibility, cwd-derived directories, tree branching, labels, custom extension entries, session search, parent sessions, and exact JSONL event vocabulary are optional.
- **Pinned Pi reference:** [`session-manager.ts`](https://github.com/earendil-works/pi/blob/dcfe36c79702ec240b146c45f167ab75ecddd205/packages/coding-agent/src/core/session-manager.ts) demonstrates a versioned header (`CURRENT_SESSION_VERSION`), typed entries, migrations, append-oriented JSONL with rewrite paths, model/reasoning changes, tool-result messages, compaction entries, context reconstruction, and open/continue behavior. The much larger [`agent-session.ts`](https://github.com/earendil-works/pi/blob/dcfe36c79702ec240b146c45f167ab75ecddd205/packages/coding-agent/src/core/agent-session.ts) is reference-only orchestration and should not define Job Radar’s interface.
- **Gap relationship:** A session version can only faithfully encode capabilities whose structured state has been decided. Controlled context management also needs a defined relationship to complete durable state. Persistence can remain an application concern around ephemeral conversations; no evidence currently requires `AgentConversation` itself to own storage.
- **Verification evidence needed later:** Deterministic temporary-storage evidence must show versioned round-trip, restart/load/continue behavior, malformed or unsupported data handling, replay opacity/redaction, and the exact provider request after continuation. Migration and crash-safety fixtures depend on the accepted format. No live service acceptance is needed.

### 7. Controlled context management

- **Current Rust support:** The core replays every completed pair on every turn. It records token usage but has no model context-window metadata in the caller-facing `Model`, budget calculation, selection, trimming, summarization, compaction, or checkpoint state. Provider length completion is reported but does not alter context.
- **Missing capability/boundary:** The agreed gap is controlled continuation near context limits without silently dropping relevant work state. Selection, summarization, compaction, budget ownership, preservation rules, inspectability, failure behavior, and the separation between persisted full state and provider-visible context all remain later decisions.
- **Product requirement vs optional Pi behavior:** Continued work near context limits without silent loss is required. Pi’s numeric defaults, chars/4 estimator, LLM-authored summary template, auto-compaction trigger, mid-turn split, file-operation extraction, branch summaries, and custom compaction hooks are optional policies requiring independent validation.
- **Pinned Pi reference:** [`types.ts`](https://github.com/earendil-works/pi/blob/dcfe36c79702ec240b146c45f167ab75ecddd205/packages/agent/src/types.ts) offers a `transformContext` hook and post-turn stop/update hooks. [`compaction.ts`](https://github.com/earendil-works/pi/blob/dcfe36c79702ec240b146c45f167ab75ecddd205/packages/coding-agent/src/core/compaction/compaction.ts) separates token estimation, trigger, valid cut points, summary generation, and compaction preparation; [`session-manager.ts`](https://github.com/earendil-works/pi/blob/dcfe36c79702ec240b146c45f167ab75ecddd205/packages/coding-agent/src/core/session-manager.ts) persists compaction entries and reconstructs active context. These are policy references, not accepted Rust behavior.
- **Gap relationship:** Context management must account for whatever structured messages, tool results, provenance, context layers, and durable state earlier capabilities introduce. Persistence and context management are distinct, but their state models must be compatible. No selection or summarization policy is accepted here.
- **Verification evidence needed later:** Deterministic synthetic sessions must demonstrate the accepted limit behavior and make any omitted or transformed state observable, including structured-call integrity and restart equivalence where applicable. If a later policy uses model-generated summaries, deterministic fixtures can verify inputs and failure handling; semantic quality would additionally need an opt-in live human checkpoint.

## Dependency evidence for the roadmap ticket

- Scoped workspace and external-information probes both depend on a bounded invocation mechanism and host policy evaluation; neither justifies ambient authority.
- Resource-bearing context layers depend on the accepted result/provenance shapes of the capabilities that supply them, while basic host-instruction composition does not necessarily depend on tools.
- Session persistence must version the structured state that future capabilities actually introduce; freezing a text-only format first could create avoidable migration work.
- Controlled context management and persistence are separate gaps, but they need compatible state models so continuation never silently loses relevant work.
- The relative priority of workspace versus external information, the breadth of the first tool slice, approval durability, and every context-management policy remain decisions for issues #200 and #201 rather than conclusions of this inventory.

## Product requirements versus Pi reference summary

| Concern | Job Radar requirement | Pi behavior that remains optional/reference-only |
| --- | --- | --- |
| Tooling | Bounded multi-step calls, registered tools, structured results, cancellation, deterministic termination | Parallel mode, queues, dynamic tools, extension messages |
| Authority | No ambient access; allow/deny/approval per call | Generic block hook, extension-based permission gates |
| Files | Explicit approved workspace, safe containment, bounded reads/controlled outputs | cwd-wide/absolute paths, bash, direct coding edits, image behavior |
| External info | Provenance plus approval, size, and time limits | General web/browser/shell networking or arbitrary providers |
| Context | Distinguishable ordered host layers; domain prompts remain inputs | Pi tags, cwd/default coding prompt, auto-discovery, skills |
| Sessions | Job Radar-owned versioned structured continuation | Pi compatibility, trees, labels, branches, custom extensions |
| Context limits | Controlled continuation without silent loss | Pi thresholds, estimator, summary prompt, auto-compaction policy |

## Gaps and follow-up research

- A concrete external-information transport has not been selected, so its provider contract, threat model, legal constraints, and live acceptance cannot yet be specified beyond generic bounds.
- Approval durability across application restart is unresolved; decide it with the session schema rather than assuming approval is ephemeral or persisted.
- Workspace artifact semantics (direct write, staged patch, generated artifact, or application-owned document operation) require a later product/security decision.
- Context-window metadata and token accounting are not exposed by the current caller-facing Agent Model. A later slice must decide whether provider metadata, conservative app budgets, or both own those limits.
- Pi’s compaction summaries are model-generated and therefore nondeterministic in content. Deterministic tests can prove selection, prompts, records, and failure behavior, but semantic summary quality requires an explicit human checkpoint in an opt-in live check.
