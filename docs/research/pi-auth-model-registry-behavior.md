# Research: Pi authentication and model-registry behavior at `dcfe36c79702ec240b146c45f167ab75ecddd205`

## Summary

At the pinned commit, Pi’s coding-agent layer owns `auth.json` as a provider-keyed record of tagged API-key/OAuth credentials and owns a compiled TypeBox schema for `models.json`. `AuthStorage` eagerly loads at construction, serializes mutations and OAuth refresh with a file lock, and resolves runtime override → stored API key/OAuth → ambient provider environment; `ModelRegistry` also gives stored auth priority over `models.json` request auth, constructs synchronously from built-ins plus config, and destructively rebuilds on refresh. Job Radar intentionally keeps the compatible document concepts but strengthens reload, filesystem, and snapshot guarantees described below.

No credential file, credential/environment value, account identifier, or authenticated service was inspected. Examples name fields only.

## Findings

1. **Pinned, source-owned document shapes.** `auth.json` is a JSON object keyed by provider ID. Each value is either `{type:"api_key", key:string, env?: Record<string,string>}` or `{type:"oauth", ...OAuthCredentials}`; the reusable AI contract identifies OAuth’s stored fields as `access`, `refresh`, and `expires`. Pi has TypeScript types but **no runtime JSON schema for auth**: loading is `JSON.parse(content) as AuthStorageData`, so syntactically valid but structurally invalid data is accepted until later use. [Auth types, lines 24–36](https://github.com/earendil-works/pi/blob/dcfe36c79702ec240b146c45f167ab75ecddd205/packages/coding-agent/src/core/auth-storage.ts#L24-L36) [unchecked parse/load, lines 249–271](https://github.com/earendil-works/pi/blob/dcfe36c79702ec240b146c45f167ab75ecddd205/packages/coding-agent/src/core/auth-storage.ts#L249-L271) [Reusable credential contract, lines 18–38](https://github.com/earendil-works/pi/blob/dcfe36c79702ec240b146c45f167ab75ecddd205/packages/ai/src/auth/types.ts#L18-L38)

   `models.json` is runtime-validated by a source-owned compiled TypeBox schema. Its root is `{providers: Record<string, ProviderConfig>}`. Provider fields are optional `name`, `baseUrl`, `apiKey`, `api`, `oauth:"radius"`, `headers`, `compat`, `authHeader`, `models`, and `modelOverrides`. Model definitions require non-empty `id`; optional fields include `name`, `api`, `baseUrl`, `reasoning`, `thinkingLevelMap`, `input`, `cost`, `contextWindow`, `maxTokens`, `headers`, and `compat`. Overrides omit identity/transport fields and allow partial cost plus model metadata/headers/compat. The source additionally defines nested compatibility, routing, thinking-map, and tiered-cost schemas. [Nested and model schemas, lines 43–219](https://github.com/earendil-works/pi/blob/dcfe36c79702ec240b146c45f167ab75ecddd205/packages/coding-agent/src/core/model-registry.ts#L43-L219) [Provider and root schemas, lines 223–242](https://github.com/earendil-works/pi/blob/dcfe36c79702ec240b146c45f167ab75ecddd205/packages/coding-agent/src/core/model-registry.ts#L223-L242)

2. **`AuthStorage` API and state behavior.** Constructors are `create(path?)`, `fromStorage(backend)`, and `inMemory(data?)`; all call `reload()` immediately. Public behavior includes runtime overrides (`setRuntimeApiKey`, `removeRuntimeApiKey`), `reload`, `get`, `getProviderEnv` (defensive copy), `set`, `remove`/`logout`, `list`, `has`, `hasAuth`, value-free `getAuthStatus`, shallow-copy `getAll`, `drainErrors`, OAuth `login`, `getApiKey`, and OAuth-provider enumeration. [Construction and runtime overrides, lines 203–242](https://github.com/earendil-works/pi/blob/dcfe36c79702ec240b146c45f167ab75ecddd205/packages/coding-agent/src/core/auth-storage.ts#L203-L242) [Mutation/query APIs, lines 274–393](https://github.com/earendil-works/pi/blob/dcfe36c79702ec240b146c45f167ab75ecddd205/packages/coding-agent/src/core/auth-storage.ts#L274-L393) [login/logout and resolution, lines 395–538](https://github.com/earendil-works/pi/blob/dcfe36c79702ec240b146c45f167ab75ecddd205/packages/coding-agent/src/core/auth-storage.ts#L395-L538)

   `set`/`remove` do not overwrite from the in-memory snapshot: under the lock they parse the latest file, change only one provider, write the merged object, and replace the in-memory view with that merged result. If the prior load failed they retry reload and refuse mutation if it still fails. `reload` updates data only after a successful parse; on failure it records `loadError` and leaves the previous in-memory `data` intact—an upstream last-known view that Job Radar deliberately rejects for auth. [Reload and provider-scoped RMW, lines 249–305](https://github.com/earendil-works/pi/blob/dcfe36c79702ec240b146c45f167ab75ecddd205/packages/coding-agent/src/core/auth-storage.ts#L249-L305)

3. **Credential resolution precedence.** Coding-agent `AuthStorage.getApiKey` uses: (1) runtime provider override; (2) stored `api_key`, resolving its `key`; (3) stored OAuth, using a valid token or locked refresh; (4) provider ambient environment via `getEnvApiKey`, unless `includeFallback:false`. A stored credential therefore prevents ambient fallback even when an unknown OAuth provider or failed refresh yields no key. [Exact order, lines 465–530](https://github.com/earendil-works/pi/blob/dcfe36c79702ec240b146c45f167ab75ecddd205/packages/coding-agent/src/core/auth-storage.ts#L465-L530) The newer reusable resolver states the same ownership rule explicitly: request override first, then stored OAuth/API key, and ambient sources only when nothing is stored; failed refresh never silently falls back. [Resolver, lines 29–67](https://github.com/earendil-works/pi/blob/dcfe36c79702ec240b146c45f167ab75ecddd205/packages/ai/src/auth/resolve.ts#L29-L67)

   At model request time, registry resolution first calls auth storage with fallback disabled; only if that returns `undefined` does it resolve provider `models.json.apiKey`. Thus effective request order is runtime override → stored API key/OAuth → `models.json` provider key/reference; ordinary provider environment fallback is omitted on this path unless encoded by the provider config or stored credential. Provider-scoped `auth.json.env` overlays process environment for substitutions. Header merge order is built-in model headers, then provider headers, then model headers (later wins); `authHeader:true` adds/replaces `Authorization` last and errors if no key. [Request resolution, lines 742–795](https://github.com/earendil-works/pi/blob/dcfe36c79702ec240b146c45f167ab75ecddd205/packages/coding-agent/src/core/model-registry.ts#L742-L795)

4. **Environment and command references.** A leading `!` is an entire shell-command reference. Otherwise `$NAME` and `${NAME}` interpolate environment values, `$$` escapes `$`, `$!` escapes a literal leading bang, and other text is literal; provider-scoped `env` wins over `process.env` using truthy (`||`) semantics. A missing referenced variable makes the whole template unresolved. [Parser and environment precedence, lines 14–113](https://github.com/earendil-works/pi/blob/dcfe36c79702ec240b146c45f167ab75ecddd205/packages/coding-agent/src/core/resolve-config-value.ts#L14-L113) Commands execute synchronously with a 10-second timeout, suppress stderr, trim stdout, map errors/nonzero/empty output to unresolved, and are process-lifetime cached in normal resolution; uncached request helpers deliberately rerun them. [Execution/cache, lines 138–227](https://github.com/earendil-works/pi/blob/dcfe36c79702ec240b146c45f167ab75ecddd205/packages/coding-agent/src/core/resolve-config-value.ts#L138-L227) Status checks recognize command configuration without executing it. [Status behavior, lines 797–824](https://github.com/earendil-works/pi/blob/dcfe36c79702ec240b146c45f167ab75ecddd205/packages/coding-agent/src/core/model-registry.ts#L797-L824)

5. **Provider/model merge and override semantics.** Provider-level `baseUrl` and `compat` modify every built-in model for that provider. `modelOverrides` then applies to matching built-ins; simple fields replace, `thinkingLevelMap` is shallow-merged, cost is fieldwise merged (tiers replace), and compat is shallow-merged with an extra nested merge for OpenRouter routing, Vercel routing, and chat-template kwargs. [Merge functions, lines 297–367](https://github.com/earendil-works/pi/blob/dcfe36c79702ec240b146c45f167ab75ecddd205/packages/coding-agent/src/core/model-registry.ts#L297-L367) Custom models inherit API/base URL from model → provider → first built-in model; defaults are name=id, reasoning=false, input=text, zero cost, context 128000, max output 16384. They are upserted by `(provider,id)` after built-in overrides, so a collision replaces the entire built-in/overridden entry. [Parsing/defaults, lines 622–674](https://github.com/earendil-works/pi/blob/dcfe36c79702ec240b146c45f167ab75ecddd205/packages/coding-agent/src/core/model-registry.ts#L622-L674) [Custom-wins merge, lines 496–508](https://github.com/earendil-works/pi/blob/dcfe36c79702ec240b146c45f167ab75ecddd205/packages/coding-agent/src/core/model-registry.ts#L496-L508)

6. **Construction, reload, startup/manual refresh.** `ModelRegistry.create(authStorage,path?)` defaults to the agent directory’s `models.json`; `inMemory` disables that file. Its private constructor normalizes the path and immediately calls `loadModels`, so startup load is synchronous. [Construction, lines 375–397](https://github.com/earendil-works/pi/blob/dcfe36c79702ec240b146c45f167ab75ecddd205/packages/coding-agent/src/core/model-registry.ts#L375-L397) `refresh()` first clears request config/header/error state and globally resets dynamic API/OAuth registrations, then reloads disk configuration and reapplies extension-registered providers. [Refresh, lines 399–416](https://github.com/earendil-works/pi/blob/dcfe36c79702ec240b146c45f167ab75ecddd205/packages/coding-agent/src/core/model-registry.ts#L399-L416) Pi documents that opening `/model` reloads the file; there is no watcher. [Models documentation, line 92](https://github.com/earendil-works/pi/blob/dcfe36c79702ec240b146c45f167ab75ecddd205/packages/coding-agent/docs/models.md#L92)

   Reload is **not transactional**. State is cleared before parsing, and an invalid file yields empty custom maps/models, after which `loadModels` installs built-ins only. Consequently invalid `models.json` drops the previously valid custom registry rather than retaining last-known-good. [Load behavior, lines 425–452](https://github.com/earendil-works/pi/blob/dcfe36c79702ec240b146c45f167ab75ecddd205/packages/coding-agent/src/core/model-registry.ts#L425-L452) [Invalid-load return and validation, lines 510–620](https://github.com/earendil-works/pi/blob/dcfe36c79702ec240b146c45f167ab75ecddd205/packages/coding-agent/src/core/model-registry.ts#L510-L620)

7. **Validation and errors.** `models.json` permits JSON comments (stripped before parse), reports all TypeBox schema errors with dotted paths, then applies semantic validation: OAuth requires base URL; override-only providers need an actual override; non-built-in custom providers need base URL; API must exist at provider/model level for non-built-ins; and `contextWindow`/`maxTokens` provided on custom model definitions (not `modelOverrides`) must be positive. Parse/schema/semantic/I/O failures become a stored string containing the file path, while callers still receive built-ins. [Parsing/schema diagnostics, lines 510–569](https://github.com/earendil-works/pi/blob/dcfe36c79702ec240b146c45f167ab75ecddd205/packages/coding-agent/src/core/model-registry.ts#L510-L569) [Semantic validation, lines 571–620](https://github.com/earendil-works/pi/blob/dcfe36c79702ec240b146c45f167ab75ecddd205/packages/coding-agent/src/core/model-registry.ts#L571-L620) Auth only validates JSON syntax; reload errors are accumulated for `drainErrors`, and mutation errors throw. OAuth refresh errors are recorded; Pi reloads once in case another process refreshed, otherwise returns unavailable while preserving the expired credential. [Auth load/mutation errors, lines 249–305](https://github.com/earendil-works/pi/blob/dcfe36c79702ec240b146c45f167ab75ecddd205/packages/coding-agent/src/core/auth-storage.ts#L249-L305) [Refresh failure handling, lines 493–516](https://github.com/earendil-works/pi/blob/dcfe36c79702ec240b146c45f167ab75ecddd205/packages/coding-agent/src/core/auth-storage.ts#L493-L516)

8. **Concurrency, atomicity, permissions, and security assumptions.** The file backend creates a missing parent as `0700`, creates/rewrites `auth.json` as `0600`, and uses `proper-lockfile`: synchronous writes retry `ELOCKED` ten times; async refresh uses exponential randomized retries, a 30-second stale threshold, and compromise detection. Every read-modify-write happens while locked. [Backend, lines 53–177](https://github.com/earendil-works/pi/blob/dcfe36c79702ec240b146c45f167ab75ecddd205/packages/coding-agent/src/core/auth-storage.ts#L53-L177) Expired OAuth uses double-checked locking, re-reads the latest file, and persists rotated credentials before releasing the lock. [Refresh, lines 415–463](https://github.com/earendil-works/pi/blob/dcfe36c79702ec240b146c45f167ab75ecddd205/packages/coding-agent/src/core/auth-storage.ts#L415-L463)

   Important limitations: writes use direct `writeFileSync`, not temp-file + rename, so they are not crash-atomic; reads and models reload have no shared snapshot/lock; existing directory modes are not repaired; `models.json` has no lock/permission enforcement; and this code has no explicit symlink rejection. `realpath:false` on the sync auth lock avoids requiring canonicalization but is not a symlink defense. These are observations from the implementation, not guarantees Pi claims. [Direct writes and lock option, lines 67–126](https://github.com/earendil-works/pi/blob/dcfe36c79702ec240b146c45f167ab75ecddd205/packages/coding-agent/src/core/auth-storage.ts#L67-L126) [Unlocked models read, lines 510–517](https://github.com/earendil-works/pi/blob/dcfe36c79702ec240b146c45f167ab75ecddd205/packages/coding-agent/src/core/model-registry.ts#L510-L517)

## Accepted Job Radar deviations

These product decisions come from [Wayfind Pi-inspired agent authentication and provider configuration](https://github.com/timjonaswechler/job-radar2/issues/208); they are not claims about upstream Pi:

- Use exactly one app-data root, `agents/`, with `agents/auth.json` and `agents/models.json`; do not implement Pi project/global discovery. On macOS derive `~/Library/Application Support/de.timjonaswechler.jobradar/agents/` through the app-data abstraction.
- Support direct API-key fields, OAuth, and `$ENV_VAR` references, but reject/disable `!command` and all arbitrary command execution. Synthetic examples only.
- Load manual edits at application startup and explicit UI reload only; no watcher and no implicit `/model`-style reload.
- Reload the auth and model documents as one explicit transaction/snapshot publication: validate candidates first, then publish later-request state. Invalid `models.json` retains the last-known-good registry; invalid `auth.json` makes auth unavailable and must not retain/reuse cached secrets.
- Requests already in flight retain their resolved immutable auth/model snapshot; a successful reload affects only later requests.
- Retain provider/model concepts and merge behavior where applicable, but implement the stricter Job Radar auth semantics and surface redacted diagnostics.
- Protect repository-external storage: `0700` parent and `0600` auth/config/lock files where secrets may occur; serialize mutation/refresh; use atomic temp-write + durable rename semantics; reject symlinks for protected files/path components rather than following them.
- Keep status value-free and diagnostics redacted. Do not reveal stored secrets after submission.
- Migrate singular `agent/auth.json` to `agents/auth.json` atomically only when destination is absent; verify destination protection before deleting old data; if both exist, report a redacted conflict and do not merge.
- Portability remains architectural, but permission/runtime acceptance for this ticket is macOS-only.

The map basis for these accepted deviations was read from [Wayfind Pi-inspired agent authentication and provider configuration](https://github.com/timjonaswechler/job-radar2/issues/208) only; all upstream behavior above was independently verified against pinned primary code.

## Gaps

- Pi does not publish standalone JSON Schema assets for these files at this commit: `models.json`’s schema is TypeBox source, while `auth.json` has only compile-time types plus unchecked JSON casting. A Job Radar JSON Schema artifact therefore must be derived and versioned as Job Radar-owned compatibility documentation.
- Exact crash durability of Job Radar’s future atomic-write implementation (directory `fsync`, platform rename behavior, recovery of orphan temporary files) remains an implementation/test decision; Pi provides no reference behavior.
- Upstream does not define request snapshot guarantees across reload, symlink-hardening semantics, or transactional dual-file reload; these are intentionally Job Radar-owned acceptance requirements.
