# Pi baseline for the Rust agent port

## Summary

Pin the Rust port to Pi commit [`dcfe36c79702ec240b146c45f167ab75ecddd205`](https://github.com/earendil-works/pi/commit/dcfe36c79702ec240b146c45f167ab75ecddd205), the verified `main` head observed on 2026-07-15 (commit timestamp 2026-07-14). This commit follows the `v0.80.7` release commit and includes the relevant 64-character Codex session-ID fix. The required behavior is concentrated in Pi’s MIT-licensed `packages/ai` OAuth/auth, Codex Responses, shared Responses conversion, model metadata, and event-stream files, the concrete coding-agent `AuthStorage`, and the small `packages/agent` state machine and loop. The much larger coding-agent `AgentSession`, UI, persistence, compaction, extensions, and tool implementations are deliberately excluded from the initial port.

No credential store, auth file, environment secret, access token, refresh token, or account-linked data was inspected. Research used only public upstream source and protocol documents; examples below describe field names and semantics, never credential values.

## Findings

1. **Exact immutable baseline and source ownership** — The selected commit is `dcfe36c79702ec240b146c45f167ab75ecddd205`, whose change itself clamps Codex session/request identifiers to 64 characters. All source permalinks below use that full SHA, not `main`. The repository root license is MIT, copyright © 2025 Mario Zechner. [Pinned commit](https://github.com/earendil-works/pi/commit/dcfe36c79702ec240b146c45f167ab75ecddd205) · [LICENSE](https://github.com/earendil-works/pi/blob/dcfe36c79702ec240b146c45f167ab75ecddd205/LICENSE)

2. **Browser OAuth is Authorization Code + PKCE S256 with state and loopback callback** — Pi uses the public client identifier `app_EMoamEEZ73f0CkXaXp7hrann`, authorization endpoint `https://auth.openai.com/oauth/authorize`, token endpoint `https://auth.openai.com/oauth/token`, redirect `http://localhost:1455/auth/callback`, scopes `openid profile email offline_access`, a fresh 16-byte random hex `state`, and PKCE `code_challenge_method=S256`. Additional authorize parameters are `id_token_add_organizations=true`, `codex_cli_simplified_flow=true`, and `originator=pi`. The listener defaults to `127.0.0.1` (override `PI_OAUTH_CALLBACK_HOST`), accepts only `/auth/callback`, requires exact state match and a code, and supports a race with pasted code/full redirect URL. This matches PKCE’s first-party standard requirement that every authorization request receive a unique verifier and transformed challenge. [Pi OAuth implementation](https://github.com/earendil-works/pi/blob/dcfe36c79702ec240b146c45f167ab75ecddd205/packages/ai/src/utils/oauth/openai-codex.ts) · [Pi PKCE helper](https://github.com/earendil-works/pi/blob/dcfe36c79702ec240b146c45f167ab75ecddd205/packages/ai/src/utils/oauth/pkce.ts) · [RFC 7636](https://datatracker.ietf.org/doc/html/rfc7636)

3. **Headless device-code login is an OpenAI-specific two-stage flow, not generic RFC device authorization** — Pi POSTs JSON `{client_id}` to `/api/accounts/deviceauth/usercode`, displays `/codex/device`, polls `/api/accounts/deviceauth/token` using `device_auth_id` and `user_code` for up to 15 minutes, treats HTTP 403/404 and `deviceauth_authorization_pending` as pending, honors `slow_down`, then exchanges the returned `authorization_code` and `code_verifier` at the normal token endpoint with redirect `https://auth.openai.com/deviceauth/callback`. Browser login remains the default. Port this only as observed behavior and label it unstable/private because these paths and response shapes are not a published stable OpenAI OAuth contract. [Pi OAuth implementation](https://github.com/earendil-works/pi/blob/dcfe36c79702ec240b146c45f167ab75ecddd205/packages/ai/src/utils/oauth/openai-codex.ts) · [Pi polling helper](https://github.com/earendil-works/pi/blob/dcfe36c79702ec240b146c45f167ab75ecddd205/packages/ai/src/utils/oauth/device-code.ts)

4. **Credential shape and refresh semantics are precise** — A successful exchange/refresh must contain `access_token`, `refresh_token`, and numeric `expires_in`; Pi stores `{type:"oauth", access, refresh, expires, accountId}`, with `expires = now + expires_in*1000`. It extracts `accountId` from the access-token JWT payload claim `https://api.openai.com/auth.chatgpt_account_id`; this is payload decoding, not signature verification. Refresh POSTs form data `grant_type=refresh_token`, the current refresh token, and client ID; the entire returned credential replaces the old one, preserving refresh-token rotation and re-deriving account ID. The request auth projection is only `{apiKey: access}`. [OAuth implementation](https://github.com/earendil-works/pi/blob/dcfe36c79702ec240b146c45f167ab75ecddd205/packages/ai/src/utils/oauth/openai-codex.ts) · [Credential types](https://github.com/earendil-works/pi/blob/dcfe36c79702ec240b146c45f167ab75ecddd205/packages/ai/src/auth/types.ts)

5. **Refresh must be serialized and persisted before use** — At the reusable `packages/ai` layer, a non-expired OAuth token takes no lock. At `now >= expires`, auth resolution enters `CredentialStore.modify`, re-reads under lock, refreshes only if still expired, and persists the rotated credential before releasing the lock. The coding agent’s concrete `AuthStorage` implements the same read-modify-write shape over `auth.json`: it creates the parent directory with mode `0700`, creates and rewrites the file with mode `0600`, uses `proper-lockfile` for synchronous mutations and asynchronous refresh, merges only the changed provider into the latest on-disk JSON, and exposes status without credential values or refresh. On refresh failure it reloads once in case another process succeeded; otherwise it returns no credential while preserving the expired credential for later login/retry. Pi uses exact expiry with no proactive skew. The Job Radar map intentionally narrows Pi’s broader API-key/environment fallback behavior to stored OAuth only and additionally requires a macOS application-data path outside the repository plus redacted errors. [Reusable resolver](https://github.com/earendil-works/pi/blob/dcfe36c79702ec240b146c45f167ab75ecddd205/packages/ai/src/auth/resolve.ts) · [Store contract](https://github.com/earendil-works/pi/blob/dcfe36c79702ec240b146c45f167ab75ecddd205/packages/ai/src/auth/types.ts) · [Concrete AuthStorage](https://github.com/earendil-works/pi/blob/dcfe36c79702ec240b146c45f167ab75ecddd205/packages/coding-agent/src/core/auth-storage.ts)

6. **Current Codex registry is generated capability data, not live discovery** — At the pin, `openai-codex.models.ts` defines seven IDs: `gpt-5.3-codex-spark`, `gpt-5.4`, `gpt-5.4-mini`, `gpt-5.5`, `gpt-5.6-luna`, `gpt-5.6-sol`, and `gpt-5.6-terra`. Every entry uses provider `openai-codex`, API `openai-codex-responses`, base URL `https://chatgpt.com/backend-api`, reasoning enabled, max output 128,000, and explicit input modalities, context windows, prices/tier prices, `thinkingLevelMap`, and (except Spark) `compat.supportsToolSearch=true`. Pi exposes `off`, `minimal`, `low`, `medium`, `high`, and `xhigh` for all seven models; the three 5.6 variants additionally expose `max`. The metadata maps `minimal` to provider effort `low`, retains `xhigh`, and retains `max` only for those 5.6 variants. Unsupported requested levels are clamped to the nearest supported level, preferring the next higher level. Rust should copy the pinned metadata as versioned data and preserve unknown future fields, not infer capabilities from model-name substrings. Pi’s registry offers provider/model lookup and runtime registration around generated metadata. [Pinned Codex models](https://github.com/earendil-works/pi/blob/dcfe36c79702ec240b146c45f167ab75ecddd205/packages/ai/src/providers/openai-codex.models.ts) · [Registry behavior](https://github.com/earendil-works/pi/blob/dcfe36c79702ec240b146c45f167ab75ecddd205/packages/ai/src/models.ts) · [Model/types contract](https://github.com/earendil-works/pi/blob/dcfe36c79702ec240b146c45f167ab75ecddd205/packages/ai/src/types.ts)

7. **Request construction targets ChatGPT’s Codex Responses endpoint** — Normalize base URL by appending `/codex/responses` unless already ending in `/codex` or `/codex/responses`. The core JSON is `model`, `store:false`, `stream:true`, `instructions` (system prompt or fallback), converted `input`, `text:{verbosity:"low"}`, `include:["reasoning.encrypted_content"]`, a session-derived `prompt_cache_key` clamped to 64 characters, `tool_choice:"auto"`, and `parallel_tool_calls:true`. Optional fields are temperature, service tier, immediate tools, and reasoning `{effort, summary}` after the model’s thinking-level mapping. Tools become Responses function tools with JSON Schema and `strict:null` for Codex. User text/images become `input_text`/data-URL `input_image`; assistant reasoning, text, and function calls are replayed; tool results become `function_call_output`. IDs are sanitized/clamped and foreign tool-call item IDs are re-keyed. [Codex request builder](https://github.com/earendil-works/pi/blob/dcfe36c79702ec240b146c45f167ab75ecddd205/packages/ai/src/api/openai-codex-responses.ts) · [Shared message/tool conversion](https://github.com/earendil-works/pi/blob/dcfe36c79702ec240b146c45f167ab75ecddd205/packages/ai/src/api/openai-responses-shared.ts) · [Cache-key clamp](https://github.com/earendil-works/pi/blob/dcfe36c79702ec240b146c45f167ab75ecddd205/packages/ai/src/api/openai-prompt-cache.ts)

8. **Headers and transports are compatibility-sensitive** — Both transports send bearer auth, `chatgpt-account-id`, `originator: pi`, and a Pi user agent. SSE adds `OpenAI-Beta: responses=experimental`, JSON content type, event-stream accept, and (when present) the same clamped session ID in `session-id` and `x-client-request-id`; it may zstd-compress JSON at level 3. WebSocket uses the same `/codex/responses` URL with `wss`, `OpenAI-Beta: responses_websockets=2026-02-06`, session/request IDs, and a first frame `{type:"response.create", ...body}`. `transport:auto` tries WebSocket, retries one pre-stream connection-limit failure, then falls back to SSE only if no stream event has started; after an emitted event, transport failure is terminal to avoid duplicate output/tool calls. Session WebSockets are reusable, idle-expire after five minutes, age out after 55 minutes, and may continue with `previous_response_id` plus only the input delta while retaining `store:false`. For a minimal first Rust milestone, explicit SSE is the safer deliberate subset; WebSocket parity should be a separately tested follow-up because the beta header/date, connection cache, continuation diffing, proxy behavior, and fallback safety are substantial. [Codex transport implementation](https://github.com/earendil-works/pi/blob/dcfe36c79702ec240b146c45f167ab75ecddd205/packages/ai/src/api/openai-codex-responses.ts)

9. **Streaming translation has a stable internal event vocabulary** — Pi maps Codex `response.done`, `response.completed`, and `response.incomplete` to a terminal Responses `response.completed` event, preserving normalized status; top-level `error` and `response.failed` become errors. Shared processing translates output items into indexed blocks and emits `thinking_start/delta/end`, `text_start/delta/end`, and `toolcall_start/delta/end`, surrounded by provider-level `start` and terminal `done` or `error`. Reasoning summary/text deltas append to thinking; output/refusal deltas append to text; function-argument deltas are incrementally JSON-salvaged, but the scratch buffer is removed before persistence. Terminal output items overwrite/finalize blocks, preserve encrypted reasoning as a replay signature, preserve response/message IDs, calculate usage/cost, subtract cached/cache-write tokens from uncached input, and map statuses to stop reasons (`completed→stop`, `incomplete→length`, failed/cancelled→error; any tool call changes stop to `toolUse`). Ending without a terminal response event is an error. [Codex event adapter/SSE parser](https://github.com/earendil-works/pi/blob/dcfe36c79702ec240b146c45f167ab75ecddd205/packages/ai/src/api/openai-codex-responses.ts) · [Responses event translator](https://github.com/earendil-works/pi/blob/dcfe36c79702ec240b146c45f167ab75ecddd205/packages/ai/src/api/openai-responses-shared.ts) · [Async event stream](https://github.com/earendil-works/pi/blob/dcfe36c79702ec240b146c45f167ab75ecddd205/packages/ai/src/utils/event-stream.ts)

10. **Minimal Agent behavior for this Rust slice is smaller than either Pi’s full Agent loop or coding-agent AgentSession** — The reusable Pi Agent establishes the useful invariants: `prompt` rejects a concurrent active run, appends/emits user messages, streams a mutable partial through message updates, commits the terminal assistant message, and preserves lifecycle ordering (`agent_start`, `turn_start`, message start/update/end, `turn_end`, `agent_end`); awaited listeners settle before idle, and errors/abort still produce lifecycle completion. Pi’s full loop also supports tools, steering, and follow-up queues, but this map expressly excludes all three. The Rust slice therefore needs only a system prompt, selected model and reasoning level, an in-memory append-only user/assistant transcript, a streaming partial, cancellation/error state, and a session ID passed to the provider. Auth is resolved immediately before each provider call so expiry is checked for every turn. Branching, compaction, labels, custom entries, storage formats, extensions, and UI are not needed. [Agent state machine](https://github.com/earendil-works/pi/blob/dcfe36c79702ec240b146c45f167ab75ecddd205/packages/agent/src/agent.ts) · [Agent loop](https://github.com/earendil-works/pi/blob/dcfe36c79702ec240b146c45f167ab75ecddd205/packages/agent/src/agent-loop.ts) · [Agent event/types](https://github.com/earendil-works/pi/blob/dcfe36c79702ec240b146c45f167ab75ecddd205/packages/agent/src/types.ts) · [Optional session core](https://github.com/earendil-works/pi/blob/dcfe36c79702ec240b146c45f167ab75ecddd205/packages/agent/src/harness/session/session.ts)

12. **Compatibility and security risks are material** — (a) OpenAI auth/device and `chatgpt.com/backend-api/codex/responses` behavior is observed first-party-service behavior through Pi, not a stable public API promise; endpoints, client ID, claims, beta headers, scopes, or required originator can change. (b) Model availability is account/plan/server controlled and the generated registry can become stale. (c) JWT payload decoding must never be mistaken for authentication/verification; account ID is only request-routing metadata obtained from an already confidential bearer token. (d) Exact-expiry refresh without skew can race clock/network latency; match Pi for parity or document a deliberate skew deviation. (e) Any retry/fallback after streamed output can duplicate side effects; Pi intentionally forbids it. (f) Persist encrypted reasoning and IDs for stateless replay, but never persist temporary partial JSON or expose bearer/account headers in diagnostics. (g) Proxy, callback binding, cancellation, timeout, rate-limit reset, zstd, SSE CRLF/multiline framing, and malformed/partial events need fixture tests using synthetic tokens only. ## Deliberate exclusions

- No inspection, import, migration, display, or testing of real credential stores/auth files; no live login and no live account/model request.
- No copying of credential values, JWT payloads, account IDs, callback query strings, environment values, or account-linked diagnostics.
- No full `packages/coding-agent/src/core/agent-session.ts` port: its UI, extension hooks, prompt templates, compaction, retries, branching/navigation, session manager, settings, model selector, and persistence orchestration exceed the minimal protocol/agent objective. [High-level AgentSession (reference only)](https://github.com/earendil-works/pi/blob/dcfe36c79702ec240b146c45f167ab75ecddd205/packages/coding-agent/src/core/agent-session.ts)
- No WebSocket connection pooling/continuation in the first minimal implementation; begin with explicit SSE while retaining transport abstraction and no-retry-after-output invariant.
- No non-Codex providers, image generation, TUI/CLI login UI, browser build compatibility, provider-specific proxy implementation, pricing UI, or live model discovery.
- No claim that ChatGPT subscription OAuth is equivalent to OpenAI API-key access or that the observed endpoints are a public supported integration contract.

## Recommended Rust parity tests

Use only fabricated JWT-shaped strings containing a dummy account claim and mocked HTTP/SSE; never use a real token.

1. PKCE/state/callback validation, manual-input parsing, cancellation, and device polling status transitions.
2. Exchange/refresh required fields, expiry calculation, refresh-token rotation, account-claim absence, and redacted errors.
3. Concurrent expired-token resolution refreshes exactly once and persists before returning; refresh failure never falls back.
4. Exact request snapshot for text/history/reasoning and 64-character cache/session IDs.
5. SSE chunk boundaries, multiline `data:`, `[DONE]`, malformed JSON, missing terminal event, error/failed/incomplete events.
6. Text/thinking deltas, terminal overwrite, encrypted reasoning replay, and usage/cache accounting.
7. Agent lifecycle ordering, auth resolution per turn, multi-turn transcript replay, abort/error completion, and idle settlement. Tool execution, steering, and follow-up queues are excluded from this slice.

## Gaps

OpenAI does not appear to publish a stable public specification for the ChatGPT subscription OAuth client, device-auth routes, account claim, Codex backend endpoint, or dated WebSocket beta used here. Consequently, Pi’s pinned MIT source is the exact behavioral reference, while RFCs cover only generic OAuth/PKCE security. Before shipping, legal/product review should confirm permitted use of ChatGPT subscription OAuth and the private/beta backend; compatibility should be guarded with synthetic contract tests and an explicitly versioned upstream pin.
