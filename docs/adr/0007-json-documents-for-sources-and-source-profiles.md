# Keep sources and source profiles as authoritative JSON documents

Building on the JSON-only profile direction tracked in GitHub issue #34, Job Radar will treat Quellen and Quellenprofile as authoritative JSON documents rather than SQLite-owned domain records. This keeps source knowledge diffable, agent-editable, importable, and consistent with built-in/custom profile files. SQLite may still index these documents and store runtime state, search runs, findings, caches, and diagnostics, but it should not be the only source of truth for sources or source profiles.

## Consequences

This applies to user-authored and built-in sources as well as source profiles: source JSON files and source-profile JSON files are the durable artifacts. Sources should reference source profiles by stable profile key and selected access-path key, not by database row ID. The JSON documents should stay domain-focused: use `schemaVersion`, omit database IDs, timestamps, persisted validation errors, built-in/custom flags, and first-pass i18n metadata; profile validity, origin, and diagnostics are computed from loaded documents and their location. Custom documents do not override built-in documents: a custom source or source profile with a built-in key is reported as a duplicate-key diagnostic and ignored; users create a separate key for variants. Source status remains a domain field on source JSON documents, including built-in sources, because it controls whether a source participates in search runs. The concrete authoring target is documented in `docs/source-registry-json-model.md` and guarded by JSON Schemas under `docs/schemas/`. The database remains a runtime index/cache over loaded JSON documents and may be rebuilt in development; user-authored JSON documents must remain the durable artifacts.
