# Use a managed browser runtime for browser-based sources

Job Radar will use a pinned Chrome for Testing browser runtime for browser-based sources, downloaded into the app data directory rather than bundled directly with the app or relying on the user's system browser. The runtime download should be explicit/lazy, exposed through setup or settings, and verified with a pinned hash so source behavior stays reproducible without forcing a large browser binary into every app build. The runtime is technical infrastructure managed through files, a local manifest, and technical commands, not part of the source/browser-profile database model. Job Radar owns a runtime spec that pins the platform artifact, version, download URL, and expected hash instead of resolving the latest version at runtime; the first implementation should pin a concrete current Chrome for Testing Stable version and record the hash computed for the downloaded archive.

The system browser may remain useful as a fallback or debug option, but it is not the primary browser runtime because installed browsers differ across systems and cannot be controlled headlessly with the same reliability. The first browser-control implementation will use pinned `chromiumoxide = "0.9.1"` behind a Job Radar abstraction, so browser-based source code does not depend directly on a specific control library.

Job Radar, rather than Chromiumoxide, owns managed Chromium process creation and teardown. It launches the pinned executable with a private user-data directory (owner-only mode on Unix and the inherited protected per-user app-data ACL on Windows) and an ephemeral loopback DevTools endpoint, discovers the strictly parsed `DevToolsActivePort`, and attaches Chromiumoxide with `Browser::connect`. On macOS and Linux the browser is the leader of an owned process group; on Windows it is created suspended, assigned to a kill-on-close Job Object, and only then resumed. Native process creation and ownership establishment form one short synchronous, uninterruptible transaction; Cancellation and deadlines are checked immediately before and after it rather than claiming that the operating-system call itself is interruptible. This ensures descendant ownership exists before any cancellable endpoint or CDP wait. Graceful shutdown may fall back to synchronous process-tree termination followed by bounded whole-tree observation: Unix requires both leader reap and process-group disappearance, while Windows requires a signaled root handle and `JobObjectBasicAccountingInformation.ActiveProcesses == 0`. Dropping launch or shutdown futures and dropping an owned session run the same bounded synchronous terminate-and-observe fallback before process ownership is released. An unconfirmed tree shutdown is an infrastructure failure; native process ownership is retained in a process-lifetime quarantine without a second Drop wait crossing the caller deadline, and its session directory is atomically quarantined from runtime-status cleanup for the remainder of the process. Chromium is launched without application-owned standard-I/O pipes, so unread output cannot block teardown. Native handles and Chromiumoxide process types remain private Browser Runtime details.

`ManagedBrowserAcquisition` is the production adapter for the phase-neutral B01 interface. It resolves only the pinned installed runtime, launches through the owned process foundation, executes compiled navigation/wait/interaction instructions under caller-owned Cancellation and the Browser work boundary, and admits rendered UTF-8 bytes atomically before returning content. Every primary outcome is sealed before shutdown; cleanup-invariant loss overrides success or ordinary failure as `BrowserInfrastructureFailure`, while an existing allowance stop is not replaced by later Cancellation. The final 2,000 ms of the caller deadline remains partitioned into stage maximums of 500 ms graceful close, 1,000 ms forced termination/reap, 250 ms handler completion/abort observation, and 250 ms filesystem finalization. Each stage is capped from when it starts as well as by its absolute boundary, so early Browser work cannot enlarge an individual teardown stage. A focused environment-gated probe exercises this final interface with a deterministic local `data:` page; productive runtime administration and legacy phase callers remain on their old route until A02 migrates them.

The local runtime manifest describes only the installed state, with paths stored relative to the runtime directory under the app data directory at `browser-runtime/`; the desired target state lives in the app's runtime spec. The manifest should have its own schema version and store diagnostic/install fields such as the source download URL, archive hash, relative install directory, and relative executable path. Installation should use a versioned platform path, keep only one active installed version, download into a temporary directory, verify the archive hash, extract into a temporary location, atomically finalize the install, write the manifest only at the end, delete the archive after success, clean up clearly marked temporary install directories on later status checks, and conservatively remove older managed install directories under the same platform after a successful update. Uninstall removes the whole managed `browser-runtime/` directory because it contains only technical runtime artifacts. Runtime status distinguishes `unsupported`, `notInstalled`, `installing`, `installed`, `updateRequired`, and `invalid`; status checks stay file-system based, while a separate smoke-test command starts the current runtime through `chromiumoxide`. Progress should be reported through Tauri events with phases for `downloading`, `verifying`, `extracting`, `finalizing`, `completed`, and `failed`, while source execution and browser-profile scraping remain separate follow-up work. The implementation may use `reqwest` with Rustls, `zip`, `sha2`, `futures-util`, and `chromiumoxide` behind Job Radar modules rather than exposing those libraries to source/profile code.
